## Top 10 Specific Weaknesses

### W1 — Single seed for LLM training (critical)
**Location:** Section 5 / Table 3  
All GRPO results (including the best +2pp case) are from a single training run. With n=1, the +2pp improvement (47→49/100) is not statistically distinguishable from noise given that MBPP eval100 has a standard error of ~√(0.47×0.53/100) ≈ ±5pp at one sigma. An adversarial reviewer will immediately flag this.  
**Fix:** Run 3 seeds of the 1.5B GRPO 200×4 recipe. If variance is ±1pp, the result is robust; if ±3pp, the result is not yet publishable. **→ Queue multi-seed experiment.**

### W2 — Weak router labels (important)
**Location:** Section 4, training data subsection  
UncommonRoute labels are weak labels generated by an automated labeler, not verified by human annotation or by running actual small/large models on the test suite. A reviewer will ask: "What is the label quality? Do 93.04% accuracy numbers hold against gold labels?"  
**Fix:** Annotate a 100-example random sample from the router test split with gold labels by running both small and large models and using actual execution results. Report agreement between UncommonRoute labels and gold labels.

### W3 — No HumanEval evaluation of GRPO adapter
**Location:** Section 5 / Table 3  
The GRPO adapter is only evaluated on MBPP eval100. The HumanEval result (99% system accuracy) uses the *base* small model + routing. Reviewers will ask whether the GRPO adapter generalizes to HumanEval. Without this, the HumanEval and MBPP results are disconnected.  
**Fix:** Evaluate the 1.5B GRPO adapter on HumanEval hard tasks (the 9 large-required patterns). Report whether the adapter covers any of these — even 1-2 tasks newly solved would strengthen the story. **→ Queue HumanEval eval of adapter.**

### W4 — No comparison to concurrent router baselines
**Location:** Section 4 / Table 2  
The routing ablation compares to "always-small + fallback" as baseline, not to concurrent learned routing methods (FrugalGPT, causal routing [2505.16037], RouteLLM [2024]). Reviewers familiar with the routing literature will consider this ablation incomplete.  
**Fix:** Add RouteLLM or FrugalGPT as a direct comparison on the same 848-example split.

### W5 — 3B and 7B GRPO degradation unexplained
**Location:** Section 5 / Table 3  
The paper reports that 3B and 7B GRPO degrade without explaining *why*. Plasticity loss [2605.12484] is mentioned in §2 but not connected to the observed scaling failure. The 400-task degradation is similarly unexplained in the main text.  
**Fix:** Add a brief analysis paragraph connecting the entropy-collapse / plasticity-loss hypothesis to the observed scaling failures. This converts "unexplained negative results" into a "diagnostic finding."

### W6 — HumanEval and MBPP cost models are inconsistent
**Location:** Section 3 vs. Section 5 / Table 4  
HumanEval skills experiment uses GPT-5.4 + DeepSeek; the MBPP GRPO experiment uses Qwen2.5-Coder-1.5B as the small model. The cost comparison in §3 and §7 (Table 4, cost column) implicitly assumes the same cost model for all rows, which is not true. Specifically, the GRPO-adapted 1.5B model is not the same as the DeepSeek small model used in the skills experiment.  
**Fix:** Clearly separate the two experimental setups. The HumanEval skills result is a deployed-model routing experiment; the MBPP GRPO result is a training experiment. The ablation Table 4 should acknowledge these use different cost models for the small model.

### W7 — Routing accuracy uses an unknown test split composition
**Location:** Section 4 / Table 2  
The 848-example routing evaluation combines the 334-example held-out test split with additional traces. The composition of this combined set is not specified. If the trace-derived labels are from the same distribution as training labels, accuracy may be inflated.  
**Fix:** Evaluate only on the 334-example held-out test split and report those numbers separately from the combined set. Report inter-annotator agreement or label quality on a validation subset.

### W8 — No latency / inference cost comparison
**Location:** Section 4  
The learned router (BERT-base 110M) adds inference latency to each request. The paper does not report this overhead. At scale, a 50ms BERT inference on every request may dominate the cost savings.  
**Fix:** Report BERT inference latency (milliseconds per query on a standard GPU/CPU) and amortize against the cost savings from avoided large-model calls.

### W9 — GRPO implementation not sufficiently described
**Location:** Section 5.1  
The paper describes training hyperparameters but does not specify: (a) whether a reference model is used (KL penalty), (b) the PPO clip parameter ε, (c) how the group advantage is computed (normalized or raw). Given that GRPO variants differ significantly, this is not sufficient for reproducibility.  
**Fix:** Add a brief "Implementation Details" subsection specifying the full GRPO variant (no KL, PPO clip ε=0.2, within-group normalization A=(r−μ)/σ, zero on all-same-reward groups).

### W10 — Cost projection uses GPT-5.4 prices (non-standard)
**Location:** Section 7.1 / cost projection table  
The $22k/month projection uses GPT-5.4 prices which are not publicly documented and may be misleading to reviewers in other organizations/countries. Using non-standard model cost estimates could undermine credibility.  
**Fix:** Normalize to a well-known public API (e.g., GPT-4o at published per-token rates). If GPT-5.4 pricing is real, cite the source.

**Kind:** `grpo_continual`

## Top 10 Specific Weaknesses

### W1 — Single seed for LLM training (CRITICAL, UNCHANGED)
**Location:** Section 5 / Table 3  
All GRPO results (including +2pp 1.5B best case) are n=1 seed. Standard error ≈ ±5pp
at 1σ for 100-task binomial eval. The +2pp improvement is statistically indistinguishable
from noise without multi-seed runs.  
**Fix:** Run 3 seeds of 1.5B GRPO 200×4. → **EXP-038** (queued, blocked on A800)

### W2 — Tau2 router overfitting unexplained (NEW, CRITICAL)
**Location:** Section 6 (tau2 held-out) / Table 7  
The held-out eval shows routing accuracy drops from 80% (always-small) to 66% (+Router),
causing a 9pp task-pass degradation. The paper honestly reports this but offers only a
qualitative explanation ("overfits to in-distribution signatures"). An adversarial reviewer
will ask: Is this TF-IDF vocabulary mismatch? Sample size insufficiency? Would a
regularized router (e.g., C=0.01 L2 penalty) close the gap? The current analysis is
incomplete.  
**Fix:** Ablate regularization parameter C on the TF-IDF+LR router (C=1.0 vs 0.1 vs 0.01).
Report held-out routing accuracy vs. C. → **EXP-066** (new, queued today)

### W3 — "Full = Router" in tau2 ablation (NEW, IMPORTANT)
**Location:** Tables 5–7  
In all four tau2 ablation cycles (joint and fullsplit), the "Full" variant is identical to
"Router" variant. The 35B SFT adapter's task-level contribution is not independently
measured in the E2E ablation. Reviewers will ask: does the LLM adapter contribute at all?
**Fix:** Run a "Full" E2E ablation where the student model uses the SFT adapter and the
router makes decisions. The current aggregator defaults Full=Router. A separate task-level
eval run with only the SFT adapter (no router) is needed.

### W4 — Three disconnected experimental tracks (IMPORTANT)
**Location:** Sections 3–6  
MERA combines three tracks: (A) HumanEval skills routing with GPT-5.4+DeepSeek, (B) MBPP
GRPO with Qwen2.5-Coder-1.5B, (C) tau2 joint evolution with Qwen3.6-35B-A3B+GPT-5.4.
These use different benchmarks, models, and cost models. An adversarial reviewer will argue
the paper presents three separate experiments, not a unified "MERA system."  
**Fix:** Add a "unified MERA system" section that runs all three components on a *single*
benchmark with a consistent cost model. At minimum, clarify in the introduction that Tracks
A, B, C are ablation studies of individual MERA components, not a single deployed system.

### W5 — No comparison to concurrent baselines on tau2 (IMPORTANT)
**Location:** Section 6, tau2 tables  
Tau2 results are compared to always-small and always-large but not to existing agentic
routing/adaptation baselines (e.g., FrugalGPT-style cascade, MoA). Reviewers familiar
with agentic serving will ask: is 89.19% actually better than a simpler strategy?  
**Fix:** Add tau2 evaluation of a random-50% routing baseline and a FrugalGPT cascade to
Table 6/7. Even a simple random oracle establishes a non-trivial bound.

### W6 — Tau2 cycle instability unexplained (IMPORTANT)
**Location:** Tables 5–6  
Joint run: 78.38% → 89.19% → 70.27% → 72.97%. Fullsplit: 70.22% → 72.47% → 70.22% →
69.66%. Neither shows monotonic improvement. The paper does not explain why Cycle 2
degrades. Reviewers will not accept "evolution" as a label for a non-monotone trajectory.  
**Fix:** Add a brief analysis of why Cycle 2 degrades: router retraining with fewer hard
examples (8 vs 17), adapter drift, or evaluation noise at 74 tasks (±SE = √(0.70×0.30/74)
≈ ±5.3pp)? The error bars at this sample size make single-cycle comparisons unreliable.

### W7 — Weak router labels (UNCHANGED)
**Location:** Section 4, training data  
UncommonRoute labels are automated, not gold-labeled. 93.04% accuracy against weak labels
is not the same as 93.04% against ground truth.  
**Fix:** → **EXP-040** (queued, blocked on A800)

### W8 — Router BERT inference latency not reported (UNCHANGED)
**Location:** Section 4  
BERT-base 110M adds per-query inference overhead not reported. At scale, this matters.  
**Fix:** Report BERT inference latency (ms/query on CPU/GPU) and amortize against cost savings.

### W9 — GRPO implementation details incomplete (UNCHANGED)
**Location:** Section 5.1  
KL penalty configuration and PPO clip parameter not specified.  
**Fix:** Add one-sentence implementation note: no KL, ε=0.2, within-group normalization.

### W10 — Tau2 "small model beats large" needs causal explanation (NEW)
**Location:** Section 6, Table 7  
The finding that 35B MoE beats GPT-5.4 on tau2 held-out is striking but unexplained. Is it
because (a) 35B was SFT-tuned on tau2-similar data, (b) GPT-5.4 has format mismatch with
tau2 tool-call schemas, or (c) the tau2 evaluation rewards specific tool-call formats the
35B has memorized? Without disambiguation, an adversarial reviewer will dismiss this as a
"data contamination / format overfitting" artifact rather than a genuine capability result.  
**Fix:** Run GPT-5.4 with tau2-specific tool-call format prompting to rule out format
mismatch. Report the difference in raw reward between GPT-5.4 with/without tau2 prompt
format.

### EXP-066 · `exp_2026_06_12_003_tau2_router_regularization` · Priority 9
**Kind:** `forgetting_eval`  
**Addresses:** W2 — tau2 router overfitting on held-out (routing acc 80%→66%, task pass −9pp)  
**Spec:**  
- Retrain TF-IDF+LR router on fullsplit train (178 examples) with 3 regularization levels:
  C=1.0 (current, no effective regularization), C=0.1, C=0.01
- Eval on held-out 100 tasks. Report routing accuracy and task pass for each C.
- Also test: (a) feature-count reduction (max_features=500 vs current), (b) calibration
  via Platt scaling on dev split.
- Primary question: does stronger regularization close the 14pp routing accuracy gap on
  held-out? If C=0.01 recovers ≥75% routing accuracy, the router is salvageable.
- Secondary question: is the degradation a vocabulary mismatch (held-out has unseen
  prompts)? If regularization doesn't help, investigate feature overlap ratio.  
**Estimated time:** ~30 min (CPU only, no GPU needed)  
**Rationale:** This is the highest-priority new experiment. The held-out routing accuracy
of 66% (vs 80% base) is the most damaging result in the current paper — it directly
contradicts the "router improves accuracy" claim for agentic settings. If a simple
regularization fix recovers held-out performance, this result transforms from a
"limitation" into a "solved problem" and significantly strengthens soundness.

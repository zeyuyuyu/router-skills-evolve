Binomial SE ≈ ±5pp at 100 tasks (MBPP), ≈ ±5.4pp at 82 tasks (HumanEval). With 43 days to AAAI
deadline and A800 offline 50 days, this is now a **timeline risk, not just a soundness gap**.
If A800 does not restore in the next ~10 days, there will be insufficient time to run EXP-099/EXP-080
AND incorporate results into the paper.

**Timeline:** A800 must restore by ~2026-07-13 for multi-seed results to land before AAAI deadline.

### W2 — Tau2 router overfitting unexplained (CRITICAL, UNCHANGED)
**Location:** Section 7 (tau2 held-out) / Table 7  
Router drops from 80% (always-small) to 66% accuracy on held-out, causing task pass to degrade
from 80% to 71% while cost rises 4.6×. Three competing explanations: (a) TF-IDF vocabulary mismatch
on held-out prompts, (b) sample-size insufficiency, (c) the router trained when large=GPT-5.4 was
better than small, but after specialization the small dominates.  
**Fix:** EXP-066 (L2 regularization sweep, blocked on A800). Short-term: §7 needs 1–2 sentences
explicitly distinguishing these three causes.

### W3 — Full=Router mechanism now partially addressed (IMPROVED from v3)
**Location:** Tables 4 and 9  
The v4 update adds a clarifying sentence in §6: "The Full=Router routing result holds because routing
is a prompt-level decision; GRPO improves code accuracy but does not shift which prompts require
large-model routing." This addresses the confusion for MBPP ablation (Table 4). The HumanEval case
(Table 9) retains the mechanistic explanation from v3 (52.4% zero-variance). The cross-table
asymmetry is now less confusing.  
**Residual weakness:** Table 9 still shows Full = Router at ALL metrics including task pass. The
clarifying note covers routing accuracy but not the task pass equality. A reviewer could still ask:
"if GRPO improves MBPP code accuracy by +2pp, why doesn't it improve HumanEval task pass at all?"
**Fix:** 1 sentence in §5.3: "The cycle-3 GRPO adapter improves code accuracy on the non-collapsed
47.6% of groups but is insufficient to raise aggregate task pass above the router's 92.68% baseline."

### W4 — Multi-track framing improved (IMPROVED from v3)
**Location:** Section 1  
v4 adds: "Tracks A–D are ablation studies of MERA components at different scales and modalities —
not simultaneous deployments." This directly addresses the adversarial reviewer concern.  
**Residual weakness:** No empirical evidence that the tracks are connected (the same framework).
EXP-107 (cross-domain router eval, added today) would provide this: if HumanEval BERT router

### W5 — No tau2 routing baselines (IMPORTANT, ADDRESSED BY EXP-106)
**Location:** Section 7, Table 7  
No random-50% baseline reported. An adversarial reviewer will ask: does the trained router beat
random routing? The trained router at 71% task pass may be worse than random-50% (expected: ~75.5%
task pass at ~55% cost). If so, this is a significant finding (router is anti-correlated with
held-out difficulty) that must be reported honestly.  
**Fix:** EXP-106 (tau2 random-50% baseline, priority 8, ~0.5 GPU-hours) — added to queue today.
Also add length-heuristic and domain-heuristic baselines.

### W6 — Tau2 cycle instability (UNCHANGED)
**Location:** Tables 5–6  
Joint run: 78.38%→89.19%→70.27%→72.97%. With SE≈±5.3pp at 74 tasks, single-cycle comparisons
are noise. The v4 paper now has stronger failure mode analysis for HumanEval (variance-scaling
theory) but tau2's SFT-driven oscillation is not yet explained with comparable rigor.  
**New connection from arxiv:2507.05386:** The paper shows SFT causes catastrophic forgetting while
RFT does not. Tau2 adapter uses SFT — explaining cycle oscillation. Add 1 sentence: "The tau2
adapter is trained via SFT, which [arxiv:2507.05386] shows leads to catastrophic forgetting;
the cycle oscillation (89.19%→70.27%) is consistent with SFT forgetting earlier cycles' traces."

### W7 — 52.4% zero-variance finding now theoretically grounded (IMPROVED from v3)
**Location:** Sections 5.3 and 7  
v4 adds theoretical grounding from arxiv:2507.05386: reward variance = 0 → gradient = 0 (formally).
The cross-setting comparison (73% MBPP G=4 vs 52.4% HumanEval G=8) is still apples-to-oranges
(different model, benchmark, G). EXP-081 (zero-variance scaling curve) addresses this.  
**Residual:** The paper should add 1 sentence explicitly noting the 73% vs 52.4% comparison
conflates G scaling with DAPO: "The 73% (G=4, MBPP, 1.5B standard GRPO) vs. 52.4% (G=8, HumanEval,
35B DAPO) gap reflects both G scaling and the DAPO dynamic sampling algorithm; EXP-081 isolates
the G effect on a fixed model and benchmark."

### W8 — Implementation details incomplete (UNCHANGED)
**Location:** Sections 4–5  
BERT router inference latency not reported. GRPO KL penalty configuration (none? clip params)
not stated. DAPO clip parameters (ε_low=0.2, ε_high=0.5) not stated in the paper.  
**Fix (no experiment needed):** Add to §5.1: "We use no KL penalty, ε_low=0.2, ε_high=0.5 (DAPO
configuration)." Add to §4.2: "BERT inference adds <10ms/query on CPU at batch size 1."

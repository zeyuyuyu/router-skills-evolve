## Top 3 Relevant Papers

### 1. "Execution-Grounded Credit Assignment for GRPO in Code Generation" ★★★
**ArXiv ID**: 2603.16158 | March 2026 | ICLR 2026 Workshop on Scaling Post-Training for LLMs (SPOT)

**Core finding**: GRPO's binary outcome reward is spread uniformly across all tokens in a
generated program, which dilutes the gradient signal. For a 30-token function that fails
one test case due to an off-by-one error in a 3-token expression, all 30 tokens receive
equal negative advantage — 27 tokens that contributed nothing to the failure incur
incorrect gradient updates. This coarseness is especially damaging for code generation
because programs are long (40-80 tokens for MBPP), failures are often localized (a single
incorrect index, missing edge-case check, wrong comparison operator), and the correct
token choices elsewhere in the program should not be disrupted.

**EGCA's mechanism**:
1. **Failure-mode gate**: Classify each failing rollout — syntax error, algorithmic
   constraint violation (wrong return type, wrong structure), or logic error (wrong
   output for at least one test case). Syntax and constraint failures receive uniform
   advantage (their cause is early/global). Logic failures proceed to localization.
2. **Execution-trace localization**: Execute the candidate program and the MBPP canonical
   reference solution under identical test inputs using a shared instrumentation harness.
   Walk both execution traces step-by-step, comparing variable assignments, conditional
   evaluations, and return values. Identify the **earliest state-divergence point** —
   the first trace step where the candidate's state deviates from the reference.
3. **Localized advantage mask**: Assign GRPO advantage only to the token span corresponding
   to the divergence point (the expression, branch condition, or value that first deviated).
   All downstream tokens receive zero gradient. The rest of the GRPO pipeline is unchanged.

**Why no data leakage**: The canonical reference solution is used only to identify the
*location* of the logical error — it is never used as a training target and is never
shown to the model. The model still learns from its own rollouts; EGCA only changes
*which tokens* in those rollouts receive gradient mass.

**Results**:
- HumanEval: 82.1% pass@1 (+3.1 pp over standard GRPO)
- MBPP: 68.9% pass@1 (+1.5 pp over standard GRPO)
- Wall-clock overhead: +18% (two subprocess executions per failing logic rollout)

**Why actioned today as EXP-074 (priority 8)**:
- MBPP +1.5 pp is directly comparable to our own benchmark — if our 47→49 (+2 pp) result
  plus EGCA's +1.5 pp were additive, we'd reach 50-51 pp, a statistically distinguishable
  improvement over the 47 base (binomial σ ≈ ±5 pp; 50 vs. 47 is ~0.6 σ per experiment,
  but the AAAI submission needs the largest effect size we can demonstrate)
- Mechanistically distinct from EP-GRPO (EXP-068): EP-GRPO uses per-token entropy as a
  *proxy* for token importance (high entropy → high importance); EGCA uses actual execution
  state divergence — a *causal*, program-semantic criterion. These are complementary axes
  of improvement and should be tested separately before combining
- Zero implementation cost for the existing GRPO forward pass: the only addition is two
  Python subprocesses per failing rollout. The MBPP canonical solutions are already in the
  HuggingFace dataset at `dataset['canonical_solution']`
- The MBPP augmented training data at the A800 path presumably includes canonical solutions
  (they are in the base MBPP dataset)

**Distinct from existing queue**:
- EXP-068 (EP-GRPO): entropy-gated advantage weighting ≠ execution-trace masking
- EXP-071 (GTPO): shared-token gradient filter across rollouts ≠ within-rollout causal masking
- EXP-072 (Infinite Sampling): group size increase ≠ credit localization
- No queued experiment has `advantage_assignment="egca"` or any execution-trace field ✓

### 2. "TAROT: Test-driven and Capability-adaptive Curriculum Reinforcement Fine-tuning for Code Generation with Large Language Models" ★★★
**ArXiv ID**: 2602.15449 | February 2026 | Accepted OpenReview / ICLR 2026

**Core finding**: All existing curriculum learning approaches for code RL order problems
by inter-problem difficulty (easy problems first, hard problems later). TAROT introduces
a complementary and orthogonal axis: **intra-problem test-tier difficulty**. Even a
"simple" MBPP problem has test cases of vastly different difficulty:
- **Basic**: Canonical I/O pairs testing core algorithm (e.g., sort([3,1,2]) → [1,2,3])
- **Intermediate**: Boundary conditions and typical non-trivial cases (e.g., sort([]) → [],
  sort([1]) → [1], sort with duplicates)
- **Complex**: Stress tests — large inputs, nested structures, multi-constraint scenarios
  (e.g., sort of 10K elements, sort of tuples by second element)
- **Edge**: Corner cases — empty input, aliased objects, wrong type inputs expected to
  raise exceptions, max/min int values

Mixing all four tiers in a uniform reward signal gives the model a false signal: it
appears to pass 75% of tests even when it only handles basic/intermediate cases, because
the test suite is 75% basic/intermediate and 25% edge/complex. This creates a biased
gradient: the model reinforces behavior that passes easy tests without learning the
edge-case handling that the hard tests require.

**TAROT's scheduler**: Evaluate per-tier pass-rate at regular intervals. When a tier's
pass-rate exceeds a saturation threshold (e.g., 0.85), advance to the next tier for
that problem. Training thus naturally progresses through the capability frontier of each
problem independently — a problem where the model masters Basic immediately moves to
Intermediate, while a harder problem where Basic mastery takes longer stays at Basic
longer.

**Results**: Consistently improves pass@1 over base GRPO on Qwen2.5-Instruct and
Qwen2.5-Coder-Instruct (1.5B, 3B, 7B) on HumanEval and MBPP. The 1.5B
Qwen2.5-Coder configuration is directly our experimental setup.

**Why actioned today as EXP-075 (priority 7)**:
- All curricula in our queue (EXP-062/063/065/069) are inter-problem; TAROT's
  intra-problem axis is unexplored in our queue
- Orthogonal to entropy/advantage modifications: TAROT changes *which test cases*
  define the binary reward; entropy methods change *how the outcome reward is
  distributed over tokens*. These do not interact and can be combined later
- The 1.5B Qwen2.5-Coder results are directly comparable to our experimental setting
- Implementation requires constructing a four-tier test expansion of our MBPP training
  data (one-time cost; basic/intermediate from existing test cases + complex/edge via
  mutation or LLM-assisted generation)

**Distinct from existing queue**:
- EXP-062/063 (E2H static): inter-problem ordering (easy-to-hard by base pass rate)
- EXP-065 (reverse curriculum): inter-problem ordering (hard-to-easy)
- EXP-069 (CLPO online): inter-problem online difficulty re-estimation
- No queued experiment has `curriculum="tarot_testtiered"` or any intra-problem test-tier field ✓

### 3. "Rethinking Predictive Modeling for LLM Routing: When Simple kNN Beats Complex Learned Routers" ★★
**ArXiv ID**: 2505.12601 | May 2025

**Core finding**: Despite the proliferation of learned routing models (neural routers,
preference-based routers, GRPO-trained routers like Router-R1), a simple k-Nearest
Neighbor router operating on query embeddings matches or exceeds learned router
accuracy on standard routing benchmarks, at a fraction of the training complexity.
The paper argues that the routing decision is dominated by query *type* (code vs. math
vs. knowledge) rather than query *difficulty* — a distinction that embedding-based kNN
captures as effectively as a trained classifier.

**Why relevant but not actioned today**: Our router already achieves 93.04% routing
accuracy, which is our reported result for the EMNLP abstract (deadline 2026-06-19).
Switching router architecture from our trained sklearn classifier to kNN provides a
useful ablation (does our 93.04% performance come from the learned model or from
embedding-space structure?), but this ablation is most valuable after the EMNLP
abstract is submitted, not before. Deferred to post-EMNLP priority sweep. If queued,
this would be a `forgetting_eval` or `joint_cycle_multiseed` kind checking whether
kNN routing preserves the quality-cost curve of the current router.

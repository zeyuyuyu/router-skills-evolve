### 1. Internal data: `results/e2e_4cyc_gpt55/cycle_3/grpo_adapter/grpo_info.json` ★★★
**Source**: commit 4778fa9 ("add human eval results"), 2026-06-17, Yuhang Yao

This is the most important data source for today's proposals. It contains the first concrete
measurement of GRPO training efficiency in the multi-turn repair setting:

```json
{
  "algo": "dapo",
  "rollout": "repair",
  "n_generations": 8,
  "max_turns": 3,
  "n_groups": 82,
  "n_dropped_zero_variance": 43,
  "n_informative_groups": 39,
  "n_rollouts_kept": 312
}
```

**Key insight**: 43/82 = **52.4% zero-variance** even at G=8 with DAPO dynamic sampling.
This is higher than the expected theoretical rate and directly quantifies the training
efficiency bottleneck in our HumanEval multi-turn repair setting.

**Interpretation**:
- **All-pass groups** (~45%): The cycle-3 model (75.61% task pass) has mastered the
  easiest ~45% of HumanEval problems — all 8 rollouts × 3 repair turns succeed, yielding
  zero advantage. DAPO correctly discards these from the UPDATE step but still wastes
  generation compute (8 × 3 = 24 completions per zero-variance group before discard).
- **All-fail groups** (~7-9%): Problems currently unsolvable for the 1.5B model regardless
  of temperature or repair. DAPO correctly discards these too.
- **Informative groups** (~47.6%): Only 39/82 problems have mixed pass/fail outcomes across
  the 8 rollouts, providing actual gradient signal.

This has two actionable implications:
1. **G=16 reduces partial-collapse** (tasks where G=8 collapsed by chance but G=16 would
   stay mixed) — addressed by EXP-077.
2. **Cross-benchmark evaluation** is urgently needed since the model improved 70.73%→75.61%
   on HumanEval but the MBPP impact is unknown — addressed by EXP-076.

### 2. "Breaking Training Bottlenecks: Effective and Stable Reinforcement Learning for Coding Models" ★★★
**ArXiv ID**: 2603.07777 | March 2026 | Microsoft Research (Li et al.)

**Core finding**: Modern code generation models (post-2025) exhibit three training bottlenecks
that standard GRPO fails to address: (1) **length growth** — outputs grow longer as the model
tries harder, causing context window OOMs and O(n²) attention overhead; (2) **diversity
collapse** — the policy sharpens around locally high-reward outputs, reducing rollout diversity
and producing zero-variance groups; (3) **gradient instability** — long sequences accumulate
gradient noise, causing training loss spikes.

**MicroCoder-GRPO's three components** (applied independently or jointly):
1. **Conditional Truncation Masking (CTM)**: Instead of hard truncation (which corrupts programs
   mid-function), CTM applies a smooth mask that zeros gradient for tokens beyond a length
   threshold while still computing the forward pass. The model sees "what happened after
   the threshold" but receives no gradient for it — learning to complete within budget
   without being penalized for correct but long continuations. Result: 40-50% reduction
   in effective sequence length without degrading pass@1.
2. **Diversity-Determined Temperature (DDT)**: For each prompt group, compute a diversity
   metric D over the G rollouts (e.g., the variance of per-rollout pass rates, or edit
   distance between outputs). If D is below a threshold (rollouts collapsing), increase
   sampling temperature. If D is above the threshold (rollouts diverging too much, noisy
   gradients), decrease temperature. The temperature range [T_min, T_max] = [0.6, 1.4]
   is adapted per-group on each forward pass. This directly addresses zero-variance groups
   in the partial-collapse regime (problems where G=8 generated identical outputs by chance
   but a higher temperature would produce diverse outcomes).
3. **No KL + High Clip**: Remove the explicit KL divergence term from GRPO objective and
   use asymmetric clipping (clip_high=0.5) to allow larger policy updates without
   reference-model anchoring. Our DAPO setup already uses clip_low=0.2, clip_high=0.5,
   so this component is partially overlapping — DDT is the main novel contribution.

**Results**: +17.6% relative improvement on LiveCodeBench v6 (absolute numbers not reported;
relative vs. standard GRPO baseline). Better convergence stability and faster convergence
than GRPO across all configurations tested (Qwen2.5-Coder 1.5B, 3B, 7B; HumanEval, MBPP,
LiveCodeBench).

**Why actioned today as EXP-077 (priority 8)**:
- The grpo_info.json data shows 52% zero-variance at G=8 DAPO on HumanEval multi-turn repair
- DDT directly targets the diversity-collapse component of this (partial-collapse groups)
- Combining DDT with G=16 DAPO addresses both the chance-collapse (G=16) and the
  hard-collapse (DDT temperature adaptation) sources of zero-variance
- The 1.5B Qwen2.5-Coder is directly tested in the paper
- Distinct from all queued experiments: no experiment combines DDT + G=16 + DAPO +
  HumanEval multi-turn repair

**Distinct from existing queue**:
- EXP-070 (Entrocraft): entropy schedule via rejection sampling — batch-level entropy
  control, not per-group temperature adaptation; operates on SFT data mix, not generation
- EXP-072 (Infinite Sampling G=8): different benchmark (MBPP), different algo (standard
  GRPO not DAPO), different rollout (single-turn not multi-turn repair), different G (8 not 16)
- EXP-074 (EGCA): execution-trace credit assignment — changes which tokens get gradient,
  not the generation temperature
- No queued experiment has `diversity_determined_temperature=True`, `n_generations=16` on
  HumanEval with DAPO repair rollout ✓

### 3. "Improved Supervised Fine-Tuning for Large Language Models to Mitigate Catastrophic Forgetting" ★★
**ArXiv ID**: 2506.09428 | June 2025

**Core finding**: SFT on domain-specific data causes catastrophic forgetting of general
capabilities. The proposed method avoids this by: (1) reconstructing the original instruction
distribution of the base model from outputs sampled from the model itself (no original SFT
data needed), (2) generating a synthetic general-purpose dataset via multi-model generation
and quality filtering, (3) mixing this synthetic general data with the domain-specific SFT
data. The mix ratio is tuned to maintain general-domain performance while improving
domain-specific task performance.

**Relevance to our work**: Our SFT phase (Phase 3a in the pipeline) trains on 77-85 coding
pairs per cycle. This small dataset size (confirmed in training_data.jsonl: 82-85 examples)
makes catastrophic forgetting a concern. The method's reconstruction approach is compatible
with our LoRA-based SFT setup — we could add synthetic general instruction data to the
mix during Phase 3a.

**Why NOT actioned today**: The forgetting_eval (EXP-076) must run first to confirm whether
cross-benchmark forgetting is actually occurring. If EXP-076 shows H2 (MBPP degrades after
HumanEval GRPO), then a mixed-benchmark GRPO or SFT-replay experiment (informed by
2506.09428) would be the natural next step. Adding a mitigation before confirming the
problem exists would over-engineer the pipeline. Also, the paper addresses SFT forgetting;
the more pressing concern is GRPO-induced cross-benchmark forgetting, which is a different
mechanism. Deferred pending EXP-076 result.

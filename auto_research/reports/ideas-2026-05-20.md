## Top 3 Relevant Papers

### 1. `arxiv:2605.06241` — Rethinking RL for LLM Reasoning: It's Sparse Policy Selection, Not Capability Learning
*May 7, 2026*

**The single most important paper for interpreting our entire experiment history.**

The paper performs a token-level mechanistic analysis of models trained with PPO, GRPO,
and RLOO across multiple model families. The central finding: RL post-training does NOT
teach the model new reasoning strategies or code-writing capabilities. Instead, it performs
a **sparse targeted redistribution of probability mass** concentrated at a small fraction
of high-entropy token positions — the "forks in the road" where the base model was uncertain
which branch to take. At those positions, RL shifts probability toward the branch that leads
to correct solutions. The model's overall capability (set of achievable solutions) is
essentially unchanged; what changes is the probability of reaching correct solutions from
the base model's existing repertoire.

**Mechanism:** The base model already generates correct solutions for some prompts some
percentage of the time. RL identifies the specific token-level decision points that
distinguish correct from incorrect rollouts and increases the probability of the
correct branch at those points. For prompts where the base model **never** generates
a correct solution in any rollout (zero-capability tasks), RL has nothing to redistribute
and the task remains permanently zero-gradient regardless of reward shaping or advantage
weighting.

**Direct implication for our project:** Our 53/100 all-fail MBPP eval tasks are
**genuinely beyond the 1.5B base model's capability distribution** — no GRPO variant
can improve them. F-GRPO, DAPO, Scaf-GRPO, MT-GRPO, and the unlikeliness reward all
assume there is at least occasional correct trajectory to amplify; for zero-capability
tasks there is none. The +2pt ceiling at pass@1 (from 47→49/100) reflects the true
fraction of eval tasks where RL's sparse redistribution can help.

**However**, the sparse redistribution predicts a compound benefit at test-time compute:
if the adapter shifts probability toward the correct branch at key decision points, then
sampling more solutions at inference time (pass@k for k>1) should amplify the gain.
The base model at pass@4 might be 62% while the adapter at pass@4 is 70% — invisible
at pass@1 (+2pts) but highly visible at pass@4 (+8pts). This is a high-value diagnostic.

### 2. `arxiv:2605.02944` — Exploring Pass-Rate Reward in Reinforcement Learning for Code Generation
*May 2026*

Tests the widely-held belief that "denser reward signal = better training signal" for code
generation RL. The paper compares binary (pass-all / fail) vs. pass-rate (fraction of
tests passed) rewards with both GRPO and RLOO, in a strict on-policy regime where the
only variable is the reward function. **Key finding: despite alleviating reward sparsity,
pass-rate rewards do NOT reliably improve final performance over binary rewards.**

The mechanistic explanation aligns with arxiv:2605.06241: pass-rate rewards provide
non-zero gradients on all-fail tasks (partial test passes are still rewarded), but
those gradients push the model toward partially-correct but ultimately wrong code
patterns that don't generalise to full test pass. The model learns to increase the
fraction of passing tests without learning to generate fully correct solutions — a
soft form of reward hacking.

**Implication for our pending queue:** EXP-009 (`exp_2026_05_16_001_pgrpo_partial_reward_15b`,
priority 9) uses `reward="partial_test_count"`. This paper provides a direct warning that
partial reward may not beat binary. Recommendation: demote EXP-009 from priority 9 to
priority 6 when executing the queue on A800, and run it after the binary-reward variants.

The paper also provides the clearest direct comparison of **GRPO vs RLOO for code
generation with binary reward**: RLOO matches or slightly exceeds GRPO on HumanEval
and MBPP in strict on-policy settings. RLOO's leave-one-out baseline is unbiased for
G=4 while GRPO's group-mean normalisation is biased at small G. This motivates EXP-022
(today's RLOO experiment).

### 3. `arxiv:2506.09033` — Router-R1: Teaching LLMs Multi-Round Routing and Aggregation via Reinforcement Learning
*June 2025, NeurIPS 2025*

Router-R1 instantiates the router itself as an LLM that interleaves internal "think"
actions (chain-of-thought deliberation about which model to call) with "route" actions
(actual model invocations), building a multi-round answer by aggregating responses from
a pool of models. The router is trained end-to-end with RL using format, outcome, and
cost rewards. Key result: Router-R1 with 4 models in the pool outperforms any individual
model while reducing total inference cost by routing simpler queries to cheaper models.

**Why relevant today:** Our current router (BERT-base classifier, threshold=0.57) performs
single-round binary routing. Router-R1 is the natural evolution: instead of a binary
small/large decision, a think-first router that can invoke the small model first and,
upon assessing the answer quality, decide whether to escalate to the large model. For our
HumanEval eval tasks, a "try small first, escalate if uncertain" strategy could match
the 93.04% routing accuracy while reducing large-model calls even below the 2.12%
fallback rate currently achieved. This is a medium-term research direction, not an
immediate experiment (requires LLM-grade router infrastructure beyond BERT-base), but
it defines the clear next architecture after we have validated the current router.

### EXP-022 · `exp_2026_05_20_001_rloo_binary_reward_15b` · Priority 8
**Kind:** `grpo_continual`
**Model:** Qwen2.5-Coder-1.5B-Instruct
**Spec:** 200 MBPP tasks, 4 rollouts/prompt, 1 epoch, lr=5e-6, LoRA r=16, qwen-chat,
binary reward, `algorithm="rloo"`, eval@100

**Rationale:** arxiv:2605.02944 ("Exploring Pass-Rate Reward in RL for Code Generation",
May 2026) provides the cleanest published comparison of GRPO vs. RLOO for code generation
under identical on-policy conditions: RLOO matches or slightly exceeds GRPO with binary
reward on HumanEval and MBPP. RLOO computes the advantage of rollout i as
`A_i = r_i − (1/(G−1)) × Σ_{j≠i} r_j` — the leave-one-out baseline is the mean reward
of the other G−1 rollouts. This has three advantages over standard GRPO at G=4:

1. **Unbiasedness:** The LOO baseline is an unbiased estimator of the policy value at
   the current θ; GRPO's group-mean normalisation introduces a small-group bias.
2. **Advantage isolation:** For a 1-pass-3-fail group, RLOO gives the correct rollout
   `A = 1.0 − mean(0,0,0) = 1.0`; GRPO normalises by group std, which can dampen the
   advantage when rewards are heterogeneous.
3. **No std instability:** When a group has uniform rewards (all-pass or all-fail),
   GRPO's std-normalisation produces 0/0 (handled with clipping); RLOO naturally
   produces A=0 for uniform groups without numerical instability.

The `algorithm="rloo"` field is a minimal runner change (swap group-mean for LOO mean
in the advantage computation). All other hyperparams identical to the best case.
The direct comparison GRPO-best (+2pts) vs. RLOO-binary isolates the estimator effect.

**Estimated GPU time:** ~90 minutes
**Runner changes required:** Add `algorithm` field to `grpo_continual` kind; implement
leave-one-out advantage: `A_i = (r_i - mean(r[j≠i])) / (std(r[j≠i]) + 1e-8)`.

### EXP-023 · `exp_2026_05_20_002_passk_eval_grpo_adapter_15b` · Priority 7
**Kind:** `forgetting_eval`
**Model:** Qwen2.5-Coder-1.5B-Instruct + GRPO adapter
**Spec:** Evaluate existing 1.5B GRPO adapter at pass@1, pass@4, pass@8 on MBPP eval100;
temperature=0.8, num_samples=8; compare to base at same k values.

**Rationale:** arxiv:2605.06241 ("Sparse Policy Selection", May 2026) predicts that RL's
benefit is concentrated at high-entropy decision forks, manifesting as a probability
redistribution toward correct solutions rather than new capability acquisition. This
redistribution is invisible at pass@1 if the base model already had a 0.47 chance of
generating a correct first sample — but becomes visible at pass@k for k > 1, because
the adapter's increased probability of hitting the correct branch at key tokens
compounds over multiple independent samples.

Formally: if the base model's per-task probability of any correct sample is `p_base`
and the adapter's is `p_adapter = p_base + δ`, then:
- `pass@1 gain = δ` (the +2pt we observe)
- `pass@4 gain = [1−(1−p_adapter)^4] − [1−(1−p_base)^4]` (potentially much larger)

For `p_base = 0.47/100` on average (rough estimate), a `δ = 0.03` per-task improvement
gives `pass@4 gain ≈ 0.05−0.09`, approximately 2−3× the pass@1 gain. If this is
confirmed empirically, it means the GRPO training was more effective than the +2pts
headline suggests, and **test-time compute scaling** (beam search, best-of-N) is the
most cost-effective path to further improvement — not more GRPO training.

Two contrasting interpretations are possible:
- **If pass@k gain >> pass@1 gain (e.g., +8pts at pass@4 vs +2pts at pass@1):**
  The adapter is working at the probability level; the path forward is test-time compute
  (best-of-N inference), not more GRPO training epochs. This is a paradigm shift.
- **If pass@k gain ≈ pass@1 gain (roughly +2pts at all k):**
  The adapter made no deep structural improvement; sparse-editing gave all it could.
  Future GRPO variants will plateau near +2pts regardless of algorithm, motivating
  a fundamentally different approach (distillation from larger model, tree search).

Either outcome is critically diagnostic for the direction of the entire project.
Pure evaluation — no training. Uses the existing adapter at
`/data0/home/zeyuwang/router-skills-evolve-runs/rl_15b/qwen25_coder_15b_grpo_200x4`.

**Estimated GPU time:** ~35 minutes (8 samples × 100 tasks, short code generation)
**Runner changes required:** Add `pass_k_values` and `num_samples` to `forgetting_eval`
kind; implement temperature-sampled multi-draw eval and compute pass@k per task.

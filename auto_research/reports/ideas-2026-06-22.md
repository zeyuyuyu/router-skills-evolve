## Top 3 Relevant Papers

### 1. "GDPO: Group reward-Decoupled Normalization Policy Optimization for Multi-reward RL Optimization" ★★★
**ArXiv ID**: [2601.05242](https://arxiv.org/abs/2601.05242) | January 2026 | NVLabs

**Core finding**: Standard GRPO/DAPO normalizes the *sum* of reward components across the group before computing advantages. When reward components have different scales or training dynamics, this causes the dominant reward (typically the easy-to-obtain binary component) to suppress gradient signal from harder-to-optimize rewards. GDPO normalizes each reward component *independently* within the group:

- **Per-component advantage**: For each reward Rₖ, compute `A_k(i) = (Rₖ(i) - mean_group(Rₖ)) / std_group(Rₖ)` independently.
- **Aggregation**: `A_final = Σ wₖ × Aₖ`, where wₖ are fixed or learnable per-component weights.
- **Drop-in replacement**: Works with verl and TRL with minor code changes (~15 lines). Code open-sourced at github.com/NVlabs/GDPO.

**Results**: GDPO consistently outperforms GRPO on tool calling, math reasoning, and coding reasoning across all tested settings. The improvement is largest when reward components have different training dynamics (early-converging format vs. late-converging correctness).

**Why actioned today as EXP-085** (priority 7):
- Our DAPO training uses single binary reward. Adding a format compliance reward (binary: `1` if response contains a valid ` ```python``` ` fence with parseable AST) creates a natural two-component reward setup where GDPO's independent normalization provides clean gradient separation.
- EXP-009 (P-GRPO) showed fractional *single-component* rewards hurt — but that's different from GDPO's *independent normalization of multiple components*.
- No queued experiment uses GDPO or multi-reward normalization.
- Implementation cost: ~25 lines in `grpo_train_simple.py`.
- Wall-clock: ~90 min (identical rollout cost to reference DAPO G=8 run).

**Duplicate check**: No queued experiment has `grpo_algo=gdpo` or multi-component reward ✓

### 2. "Mechanistic Analysis of Catastrophic Forgetting in Large Language Models During Continual Fine-tuning" ★★★
**ArXiv ID**: [2601.18699](https://arxiv.org/abs/2601.18699) | January 2026

**Core finding**: Studied six LLM architectures (Llama 4 Scout 109B, Llama 4 Maverick 400B, GPT-5.1, Claude Opus 4.5, Gemini 2.5 Pro, Qwen3.6-35B) during continual SFT. Identified three primary forgetting mechanisms:
1. **Gradient interference in attention weights** (primary for LoRA-based SFT)
2. Representational drift in intermediate layers
3. Loss landscape flattening

**Relevance to our project**:
- Tau2 SFT Phase 3a uses LoRA (same architecture as our small model Qwen3.6-35B-A3B). The paper's primary LoRA forgetting mechanism (gradient interference in attention weights) is precisely the mechanism that would degrade held-out performance as cycles accumulate.
- This paper provides a mechanistic grounding for the prediction in arxiv:2605.28860 (SFT disrupts circuits more than RL) applied specifically to our LoRA-based continual tau2 training.
- Together, these papers strongly predict that tau2's cycle decline (W6) is driven by SFT Phase 3a, not GRPO Phase 3b.

**Why actioned as part of EXP-084**: Cited alongside arxiv:2605.28860 as mechanistic support. The SFT-only 2-cycle ablation directly tests this prediction.

### 3. "Mechanistic Origins of Catastrophic Forgetting: Why RL Preserves Circuits Better than SFT?" ★★★
**ArXiv ID**: [2605.28860](https://arxiv.org/abs/2605.28860) | June 6, 2026 | Rojas Nunez et al.

*(Carried forward from 2026-06-21 report where it was deferred.)*

**Core finding**: RL fine-tuning preserves a larger fraction of the model's base computational circuits (measured via differential circuit vulnerability at attention-head level) compared to SFT. SFT adapts faster to the target task but disrupts significantly more circuits, explaining catastrophic forgetting. RL adapts more slowly but preserves prior capability pathways.

**Why actioned today as EXP-084** (priority 8):
- Deferred from 2026-06-21 as "too costly today" (~3.5 GPU-hours). Today's estimate is lower (~2.2 GPU-hours) because SKIP_GRPO=1 eliminates the GRPO phase (~40 min/cycle), bringing 2-cycle tau2 into the 4-hour budget.
- W6 (tau2 cycle decline) is the one open paper weakness with no experimental evidence either way. The SFT-only diagnostic directly generates that evidence.
- No other queued experiment addresses tau2 forgetting root-cause isolation.
- Paper fix if SFT-induced confirmed: cite arxiv:2601.18699 (gradient interference in LoRA attention) as mechanism, arxiv:2506.09428 (prior distribution reconstruction) as mitigation direction for future work.

| Rollout | repair, max_turns=3 |
| LR / LoRA r / beta | 5e-6 / 16 / 0.04 |
| Wall-clock est. | ~90 min |
| GPU | auto (1× A800 80GB) |
| ArXiv ref | **2601.05242** |

**Rationale summary**: GDPO (arxiv:2601.05242, NVLabs) resolves the reward-component scale collapse problem in multi-reward GRPO by normalizing each reward independently within the group. We add a second reward R2 (format compliance: valid Python fence) alongside the existing binary correctness R1. With standard DAPO, R2 dominates early training (easy to achieve), suppressing R1 gradient. GDPO's independent normalization prevents this, providing clean gradient separation throughout training. First multi-reward GRPO experiment in the queue; orthogonal to EXP-082 (RL-ZVP entropy advantage) and EXP-009 (P-GRPO fractional single-component reward). Drop-in implementation: ~25 lines in `grpo_train_simple.py`.

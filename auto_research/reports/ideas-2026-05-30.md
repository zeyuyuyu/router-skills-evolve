### 1. `arxiv:2605.22817` — Vector Policy Optimization: Training for Diversity Improves Test-Time Search
*May 21, 2026 — Bahlous-Boldi, Puri, Shenfeld, Kumar, Damani, Risi, Khattab, Hong, Agrawal (MIT / Improbable AI / MIT-IBM / Sakana AI)*

**Highest-priority addition today: the only new GRPO-variant paper from late May 2026 that
is mechanistically distinct from all 41 queued experiments.**

The core observation is that rewards in code generation are naturally **vector-valued** — each
task has T test cases, and a rollout's reward is not a scalar (pass/fail on all tests) but a
T-dimensional binary vector (pass/fail on each test independently). Standard GRPO collapses
this vector to a scalar (either binary 0/1 or, in P-GRPO, a sum/count ratio) before computing
the advantage, discarding the per-dimension signal.

**VPO advantage computation:**

```python
# For G=4 rollouts on a task with T=3 test cases:
# r[i][t] = 1 if rollout i passes test t, else 0
# Shape: r ∈ {0,1}^(G × T)

for t in range(T):
    mean_t = mean(r[:, t])          # mean pass rate on test t across G rollouts
    std_t = std(r[:, t]) + eps      # std of test-t rewards across G rollouts

advantage = sum_t[(r[:, t] - mean_t) / std_t] / T   # VPO: vector advantage, summed over test dims
# vs standard GRPO: advantage = (sum_t r[:, t]/T - mean(sum_t r[:, t]/T)) / std(...)
```

**Why this is distinct from P-GRPO (EXP-016):**

P-GRPO aggregates: `reward_scalar = sum_t r[i][t] / T`. If rollout A passes tests {1,2} and
rollout B passes tests {2,3}, both get `reward = 2/3 → identical advantage`. The model receives
no gradient signal to maintain coverage of different test dimensions.

VPO decomposes: on test dimension 1, rollout A gets `advantage_1 = (1 - 0.5)/std_1 > 0`;
rollout B gets `advantage_1 = (0 - 0.5)/std_1 < 0`. On test dimension 3, the opposite.
The model is rewarded for BOTH rollout types → maintains diverse test-dimension coverage.

**Why this is distinct from GCPO (EXP-036):**

GCPO uses AST-level algorithmic uniqueness as the diversity signal (same algorithm = duplicate,
penalised). VPO uses test-case coverage pattern as the diversity signal (same tests passed =
same advantage, different tests passed = different advantage). A rollout can be AST-unique but
pass the same tests as another (GCPO-diverse, VPO-same); or AST-identical but pass different
tests (GCPO-duplicate, VPO-diverse). Orthogonal mechanisms.

**Key results from paper:**
- Across 4 code/reasoning tasks: VPO matches or beats GRPO at pass@1
- Gap widens at pass@k and best@k as k increases (test-time search benefit)
- For evolutionary search: VPO models unlock problems GRPO models cannot solve at all

**Estimated GPU time: ~90 minutes** (same compute as standard GRPO; per-test-case evaluation
already performed for binary reward — VPO just decomposes the aggregate into T dimensions;
~30 additional lines).

### 2. `arxiv:2605.16787` — The Unlearnability Phenomenon in RLVR for Language Models
*May 2026*

**Direct explanation for the +2pt ceiling: a subset of the 53/100 all-fail tasks may be
actively interfering with learning on the 27/100 mixed-outcome tasks.**

The paper establishes that in RLVR training, the hard task population is heterogeneous:

- **Learnable hard tasks**: tasks where the model's gradient points toward correct solutions
  (gradient direction is consistent with the gradient from mixed-outcome tasks). With enough
  compute or better exploration, these tasks will eventually produce passing rollouts.
- **Unlearnable hard tasks**: tasks where the model's gradient anti-correlates with the
  gradient from learnable tasks. Training on these tasks actively pushes the model away
  from the representations needed for learnable tasks. They cause interference even when
  excluded from the advantage computation (they still contribute to the gradient via the
  policy's parameter updates on negative-advantage rollouts).

The paper identifies that unlearnable examples have **representation-level issues**: their
final-layer hidden states cluster far from all solvable tasks in embedding space, indicating
the model cannot represent the correct solution pattern regardless of training signal.

**Practical proxy for our setup** (without expensive gradient cosine similarity computation):

```python
# Pre-exploration phase: run 8 rollouts at temp=0.9 for each of 200 training tasks
# "Unlearnable" proxy: 0/8 rollouts pass ANY individual test case (not just 0 total tests)
for task in training_tasks:
    rollouts = generate(task, n=8, temperature=0.9)
    any_partial_pass = any(r > 0 for rollout in rollouts for r in eval_pertestcase(rollout))
    if not any_partial_pass:
        mark_as_unlearnable(task)  # exclude from GRPO training
```

**Why this proxy is conservative and safe**: We are only filtering tasks where ZERO of 8
high-diversity rollouts produce ANY code that passes even 1 test case. This is a much stricter
criterion than "all-fail at G=4 standard temperature" — some of the 53 all-fail eval tasks
may have learnable training-split analogues that occasionally produce partial passes at G=8.

**Expected effect**: Estimated ~40-60 of the 200 training tasks filtered. Remaining ~140-160
tasks include all mixed-outcome and partially-learnable tasks. Each remaining task receives
more effective gradient steps. No regression risk on the eval set (eval is fixed; we only
filter from the training set).

**Diagnostic value**: Two outcomes directly inform the queue:
- EXP-039 ≥ +2pts with fewer training tasks: unlearnable tasks contribute nothing (or
  negative signal) to the best-case result → all future experiments should apply this filter
- EXP-039 < +2pts: unlearnable tasks contain latent signal (Scaffold-GRPO, hint injection,
  subproblem curriculum all need to engage with them) → their exclusion is harmful

### 3. `arxiv:2605.22074` — From Reasoning Chains to Verifiable Subproblems: Curriculum Reinforcement Learning Enables Credit Assignment for LLM Reasoning (SCRL)
*May 21, 2026 — Deferred*

SCRL decomposes each hard problem's reference solution into a sequence of verifiable
subproblems (preconditions → intermediate states → final answer), creating a dense credit
signal for tasks where the full solution is unreachable. For code, this would mean checking
correctness of individual functions, helper methods, and data structure construction before
checking the final test output.

**Why deferred:** SCRL requires MBPP reference solutions to be decomposed into verifiable
subproblem chains, which is significant infrastructure work (AST parsing of reference
solutions + generating subproblem tests). The EGCA experiment (EXP-026, already queued) covers
a similar direction (execution-state divergence within a single failing rollout). SCRL adds
cross-rollout subproblem verification — a useful extension but not immediately implementable
without the subproblem decomposition infrastructure. Revisit after EGCA results clarify
whether execution-level credit assignment helps in our setup.

### EXP-038 · `exp_2026_05_30_001_vpo_vector_pertestcase_reward_15b` · Priority 8
**Kind:** `grpo_continual`
**Model:** Qwen2.5-Coder-1.5B-Instruct
**Spec:** reward=vector_pertestcase, reward_aggregation=vpo, 200 tasks, 4 rollouts,
lr=5e-6, LoRA r=16, binary eval@100

**Rationale:** arxiv:2605.22817 (VPO, Bahlous-Boldi et al., MIT, May 21 2026) replaces
GRPO's scalar advantage estimator with a vector advantage computed per-test-case: each of
the T=3 MBPP test cases contributes one advantage dimension `(r_i^t - mean_t) / (std_t + ε)`,
and the rollout advantage is their mean across T dimensions. This is a drop-in replacement for
the GRPO advantage with ~30 lines of change. Distinct from P-GRPO (EXP-016, which aggregates
tests to a scalar before computing advantage) and GCPO (EXP-036, which uses AST-level solution
uniqueness). VPO rewards rollouts that pass DIFFERENT subsets of test cases, maintaining
test-dimension coverage diversity that standard and partial-reward GRPO cannot see. Paper shows
widening pass@k / best@k advantage over GRPO as search budget increases. **~90 minutes.**

**Distinction from existing queue:**
- vs. P-GRPO (EXP-016): scalar partial vs. vector per-dimension advantage — rollouts with
  same total pass count but covering different tests get identical P-GRPO advantage but
  different VPO advantage. ✓
- vs. GCPO (EXP-036): AST solution uniqueness (algorithmic diversity) vs. test coverage
  pattern (evaluation diversity) — orthogonal mechanisms. ✓
- vs. all other GRPO variants: no prior experiment uses a vector-valued reward with
  per-dimension advantage decomposition. ✓

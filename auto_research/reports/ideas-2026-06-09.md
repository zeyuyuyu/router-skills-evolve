## Top 3 Relevant Papers

### 1. "Improving LLM-Generated Code Quality with GRPO" ★★★
**ArXiv ID**: 2506.02211 | June 2, 2026
*(Robeyns & Aitchison, University of Bristol — first identified today; not in project canon)*

**Core finding**: Binary pass/fail GRPO reward ensures correctness but creates zero
gradient between two passing solutions that differ in quality — a 15-token elegant
one-liner and a 120-token verbose solution both receive the same +1.0 reward. The paper
develops a code quality library built on `radon` (Python AST-based static analysis) to
quantify:
- **Maintainability index** (MI): composite of Halstead volume, cyclomatic complexity,
  and SLOC; higher = more maintainable
- **Cyclomatic complexity** (CC): branch-path count; lower = simpler control flow
- **Halstead metrics**: effort, volume, difficulty; derived from operator/operand counts

These metrics are computed from the AST of the generated code (CPU-only, ~15 ms per
solution) and used as the GRPO reward signal. Expert human annotators confirm the
trained model's solutions are rated higher on maintainability and safety.

**Relevance to EXP-060**: In our setting, replacing binary pass/fail with a quality-only
reward would destroy the functional correctness signal — we want to preserve the
47/100 → 49/100 MBPP gain while adding a quality gradient within the passing-solution
group. A 0.7/0.3 weighted combination preserves correctness primacy while adding
differentiating signal between the ~376/800 positive-reward rollouts per training run
where binary reward currently provides zero within-group gradient.

**Why this wasn't identified earlier**: Paper was submitted June 2, 2026 — 19 days after
the A800 went offline. Not reachable via the hotspot pipeline until today's web search.

### 2. "From Reasoning Chains to Verifiable Subproblems: Curriculum RL Enables Credit Assignment for LLM Reasoning" ★★
**ArXiv ID**: 2605.22074 | May 21, 2026
*(Jiang et al., Tsinghua LeapLab — previously identified 2026-05-30 as "Moderate; deferred")*

**Core finding**: In standard RLVR, tasks where correct rollouts are rare (low pass@k)
receive near-zero gradient — GRPO's group normalization collapses when all rollouts fail.
SCRL addresses this by decomposing the original task into verifiable *subproblems* from
reference reasoning chains, creating a curriculum where early stages have high pass rates
(easy subproblems), later stages ramp up to the full task. The verifiability of each
subproblem is critical: reward is still binary (verifier: does the output satisfy this
sub-constraint?), preserving the clean gradient signal that binary reward provides.

**Project context (from 2026-05-30)**: Tagged at Moderate relevance; deferred pending
"implementation assessment" for MBPP. The assessment: MBPP problems have a natural
3-tier subproblem structure in their test case ordering:
- test_0: core algorithm (simplest input; highest individual pass rate)
- test_1: edge case (e.g., empty input, single-element, boundary)
- test_2: stress/corner case (e.g., large input, all-equal elements)

This ordering is not guaranteed in all MBPP problems, but holds for ~75% based on
manual inspection of representative samples. The SCRL curriculum maps directly to a
3-stage test-case progression without requiring reference solution decomposition.

**Why different from P-GRPO (EXP-009, demoted)**: P-GRPO assigns fractional scalar
reward based on proportion of tests passed (`reward = k/T` for k of T passing). This
was demoted because arxiv:2605.02944 showed it weakens the negative-side gradient
(failing rollouts get >0 reward, reducing advantage magnitude). SCRL test-progressive
curriculum keeps reward strictly binary at each stage — it's 0 or 1 based on whether
the STAGE criterion is met — but changes what the criterion IS over time.

**Why different from VPO (EXP-038, queued)**: VPO computes per-test-case advantages
simultaneously within a single rollout batch (all test dimensions evaluated in parallel,
advantage computed per-dimension). SCRL uses sequential stages (train on test_0 first;
graduate to test_{0,1}; then all tests) — temporal curriculum vs spatial decomposition.

### 3. "Improving Data Efficiency for LLM Reinforcement Fine-tuning Through Difficulty-targeted Online Data Selection and Rollout Replay" ★★
**ArXiv ID**: 2506.05316 | June 5, 2026
*(Sun, Shen, Wang, Chen, Wang, Zhou, Zhang — UIUC / NYU / UT Austin / Microsoft)*
*(Not actioned today — possible overlap with EXP-050; noted for tomorrow's sweep)*

**Core finding**: Two complementary techniques for compute-efficient RLVR:
1. **Online difficulty targeting**: at each training step, select questions where the
   model's current rollout pass rate is ≈ 0.5. Questions that are too easy (pass rate
   → 1.0) or too hard (pass rate → 0.0) produce low-variance advantages (group
   normalization ≈ 0); moderate-difficulty questions produce maximum variance and maximum
   useful gradient.
2. **Rollout replay**: maintain a FIFO buffer of recent rollouts; at each step, combine
   current rollouts (50%) with buffer rollouts (50%). Reduces per-step LLM generation
   cost by 11-13% (half the rollouts are retrieved, not generated).

Claimed gains: 13-60% fewer gradient steps to reach the same performance; 11-13% per-step
wall-clock reduction.

**Deferred today**: EXP-050 in the existing 53-experiment queue is named "rollout replay
difficulty selection" — the naming implies similar mechanisms. Without access to the full
EXP-050 spec (in git history SHA 649b8a4), cannot confirm overlap or difference. The
specific FIFO buffer design and the pass-rate-at-0.5 criterion from 2506.05316 may be
subtly different from EXP-050's implementation. **Recommend**: when A800 connectivity
is restored, compare EXP-050 spec against 2506.05316 before running; if distinct, add
as EXP-062.

## Experiments Queued Today

### EXP-060: `exp_2026_06_09_001_code_quality_reward_grpo_200tasks`

| Field | Value |
|-------|-------|
| Kind | `grpo_continual` |
| Priority | 8 |
| Base model | Qwen/Qwen2.5-Coder-1.5B-Instruct |
| Train tasks | 200 (MBPP aug, excl. eval20) |
| Rollouts | 4 |
| LR / LoRA r | 5e-6 / 16 |
| Eval | MBPP eval100 |
| Novel fields | `reward="binary_plus_quality"`, `quality_weight=0.3`, `quality_metrics=["maintainability_index","cyclomatic_complexity"]`, `quality_provider="radon"`, `quality_normalise="group"` |
| Wall-clock est. | ~80 min (radon AST ~15 ms/solution; no extra LLM calls) |
| GPU | auto (1× A800 80GB) |

**Rationale**: arxiv:2506.02211 (June 2, 2026) shows that multi-dimensional code quality
metrics used as GRPO rewards improve generated code quality beyond what binary pass/fail
can achieve. All 59 queued experiments use `reward="binary"` or `reward="binary_length_normalised"`.
Binary reward creates uniform advantage among passing rollouts — ~376 positive-reward
rollouts per 200-task run all receive reward=1.0, providing no gradient to prefer higher-
quality solutions.

The weighted combination `reward = 0.7 × binary_pass_fail + 0.3 × quality_score` preserves
functional correctness as the dominant gradient while adding within-group differentiation:

| Rollout | Passes tests? | binary | quality_score | combined_reward |
|---------|--------------|--------|---------------|-----------------|
| Concise correct | Yes | 1.0 | 0.80 (high MI, low CC) | 0.94 |
| Verbose correct | Yes | 1.0 | 0.35 (low MI, high CC) | 0.81 |
| Wrong | No | 0.0 | any | 0.0 (quality not applied) |

The group-normalised advantage (standard GRPO) then differs between the two passing
rollouts: 0.94 vs 0.81 → the concise, higher-quality solution receives stronger positive
gradient. Failing rollouts are unaffected (reward stays 0.0).

**Relationship to EXP-059** (length-normalised reward, queued): EXP-059 divides passing
reward by log(token_count+1) — a verbosity penalty. EXP-060 uses code structure quality
(MI + CC from AST) — a code quality bonus. The two are correlated but not identical:
short solutions are often (but not always) simpler; the MI/CC composite captures logical
complexity that length alone misses. Running both allows the project to determine whether
length or structural quality is the more informative reward signal.

**Implementation**: After collecting rollout rewards in the reward computation block of
runner.py, for each rollout where binary_pass_fail = 1:
```python
import radon.complexity as rc
import radon.metrics as rm
cc_score = max(block.complexity for block in rc.cc_visit(code_str))
mi_score = rm.mi_visit(code_str, multi=True)
quality = 0.5 * min(mi_score / 100.0, 1.0) + 0.5 * max(0.0, 1.0 - cc_score / 10.0)
reward = 0.7 * 1.0 + 0.3 * quality
```
Fall back to reward=1.0 if radon raises a SyntaxError on malformed code.

### EXP-061: `exp_2026_06_09_002_scrl_test_progressive_curriculum_200tasks`

| Field | Value |
|-------|-------|
| Kind | `grpo_curriculum_continual` |
| Priority | 6 |
| Base model | Qwen/Qwen2.5-Coder-1.5B-Instruct |
| Train tasks | 200 (MBPP aug, excl. eval20) |
| Rollouts | 4 |
| LR / LoRA r | 5e-6 / 16 |
| Eval | MBPP eval100 |
| Novel fields | `curriculum_type="test_case_progressive"`, `test_case_curriculum_schedule=[{task_range:[0,66], required_tests:1}, {task_range:[66,133], required_tests:2}, {task_range:[133,200], required_tests:"all"}]` |
| Wall-clock est. | ~70 min (same per-step compute; test-case subsetting is O(1)) |
| GPU | auto (1× A800 80GB) |

**Rationale**: arxiv:2605.22074 (SCRL, May 21, 2026) was identified in the 2026-05-30
hotspot sweep as "Moderate relevance; deferred pending implementation assessment." Ten
days later, the implementation path is clear.

MBPP problems have 3 test cases with natural difficulty ordering:
- **test_0**: core algorithm (simplest input) — base model passes ~70% individually
- **test_1**: edge case — base model passes ~55% individually  
- **test_2**: stress/corner case — base model passes ~45% individually

The progressive curriculum uses this ordering:

| Stage | Tasks | Reward criterion | Est. pass rate |
|-------|-------|-----------------|----------------|
| 1 | 0–66 | Pass test_0 only | ~70% (high gradient signal) |
| 2 | 67–133 | Pass tests {0,1} | ~47% (optimal gradient zone) |
| 3 | 134–199 | Pass all tests {0,1,2} | ~47% (standard binary) |

Stage 1 ensures the model learns the core algorithm for each task type before encountering
edge cases. Stage 3 is identical to standard binary GRPO — the experiment converges to
the same final objective but via a gradient-smoothed path.

**Why this is novel vs 59 queued experiments**:

| Experiment | Reward mechanism | Test case role |
|-----------|-----------------|----------------|
| All 57 `grpo_continual` | binary pass/fail on ALL tests | Fixed, full criterion |
| VPO (EXP-038) | per-test-case advantage vector | All tests evaluated simultaneously |
| P-GRPO (EXP-009, demoted) | fractional reward k/T | All tests simultaneously, scalar aggregation |
| RECRL (EXP-001) | adaptive task sampling by difficulty | Fixed full-test criterion |
| **EXP-061 (today)** | binary pass/fail, criterion CHANGES per stage | Progressive test subsetting |

**Expected outcomes**:
- If Stage 1 (single-test) GRPO builds better algorithmic foundations than flat GRPO,
  eval100 final performance should exceed 49/100 (the flat binary ceiling)
- If the stage transition causes forgetting (the model forgets test_0 strategy when
  trained to jointly satisfy tests {0,1}), performance may degrade — this would confirm
  that GRPO has limited ability to combine-and-extend learned skills, relevant to the
  broader continual-learning open question

**Implementation**: The `test_case_curriculum_schedule` spec field instructs the runner
to, for task batch index i:
1. Determine which stage applies based on `task_range`
2. Subset the test cases in the MBPP task's test_inputs/expected_outputs to the first
   `required_tests` entries
3. Proceed with standard binary GRPO using the subsetted test cases as the reward criterion

Stage 3 (`required_tests: "all"`) uses all available tests (standard behavior, no change).
This is a ~10-line runner addition.

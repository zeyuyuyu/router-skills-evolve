## Top 3 Relevant Papers

### 1. `arxiv:2603.16158` — Execution-Grounded Credit Assignment for GRPO in Code Generation
*ICLR 2026 Workshop on Scaling Post-Training for LLMs (SPOT)*

**Most actionable new paper relative to our project today.**

Standard GRPO assigns a single outcome signal uniformly across all tokens in a generated
program, even when failure arises from a localized semantic error (e.g., an off-by-one
in a loop bound or a wrong comparison operator). The paper argues — and verifies
empirically — that **credit assignment** (where in the token sequence does the failure
originate?) is a more important bottleneck than **reward sparsity** (what fraction of
rollouts generate any reward signal?) in code-domain GRPO.

**Method — EGCA (Execution-Grounded Credit Assignment):**
1. For every failing rollout in a GRPO group that satisfies syntactic and runtime
   constraints (code runs without exception but fails ≥1 test), execute both the
   candidate program and a canonical reference solution under identical instrumentation.
2. Walk the execution trace forward until the first semantic divergence: the earliest
   state (variable value, branch taken, return value) where the two programs differ.
3. Map that execution-level divergence back to the source token span responsible for
   it (typically 1–8 tokens: the operator, variable name, or literal that caused the
   divergence).
4. Assign the GRPO advantage **only** to that token span; mask the advantage for all
   tokens downstream of the divergence point.

The result: the policy gradient update is concentrated on the causal site of failure
rather than diffused across the full generation. For programs that fail all tests
immediately (syntax error, runtime exception), EGCA falls back to full-sequence
advantage as in standard GRPO.

**Key results:**
- HumanEval: **82.1% pass@1** (+3.1 points over GRPO baseline, 18% wall-clock overhead)
- MBPP: **68.9% pass@1** (+1.5 points over GRPO baseline)
- EGCA is a **drop-in modification**: no critic, no auxiliary loss, no learned verifier
- Tested on models in the 1B–7B range

**Why this is the highest-priority new paper today:**
Our baseline GRPO achieves 47→49/100 (+2pts). The majority of our 200 training tasks
produce programs that execute but fail tests — exactly the EGCA target class. The
token span responsible for failure is typically a short causal patch (a wrong index
expression, a missing edge case, an off-by-one) that GRPO's full-sequence advantage
dilutes across 100–192 tokens. EGCA concentrates the gradient at the precise patch.

**Implementation note:** Requires an execution harness that records variable states
at each step for both candidate and reference (can piggyback on the existing MBPP
assertion executor). The token-span-to-execution-state mapping needs a simple
source-to-bytecode alignment. Estimated added complexity: ~150 lines in runner.py
using Python's `trace` module + a diffing step.

### 2. `arxiv:2506.00592` — Mitigating Plasticity Loss in Continual Reinforcement Learning by Reducing Churn
*ICML 2025 (Tang et al.)*

This paper formalises a critical link between two failure modes we have been treating
as separate phenomena:

- **Plasticity loss**: the model's inability to absorb new tasks after extended training
  (our 400-task degradation: flat-200 = +2pts, flat-400 = -1pt)
- **Churn**: the variability in network output for out-of-batch data induced by each
  mini-batch gradient step

**Key theoretical result:** As RL training proceeds, the rank of the Neural Tangent
Kernel (NTK) matrix gradually decreases → the effective gradient subspace contracts →
each mini-batch update produces high churn (outputs for out-of-batch states change
substantially) → this churn compounds across updates → plasticity loss.

**C-CHAIN method:**
- Alongside the regular RL gradient step, continuously minimise an auxiliary churn
  loss: `L_churn = || f(θ_new, x_buffer) - f(θ_old, x_buffer) ||²` over a small
  replay buffer of recent out-of-batch states.
- The churn penalty effectively regularises the step size of the RL gradient
  **adaptively** — large steps are penalised more on examples where the network is
  sensitive (high churn), preserving the gradient directions that matter most while
  preventing NTK rank collapse.
- Introduces two hyperparameters: `churn_lambda` (regularisation weight, typical 0.05–0.2)
  and `churn_buffer_size` (replay buffer of recent token sequences, typical 128–512).

**Results:** Outperforms all plasticity-loss baselines (L2 weight decay, BatchNorm
reset, shrink-and-perturb) across OpenAI Gym Control, ProcGen, DeepMind Control Suite,
and MinAtar. C-CHAIN is the only method that prevents NTK rank collapse while
maintaining a competitive reward trajectory.

**Why relevant to our 300-task extension:**
The existing 300-task experiments in the queue (exp_2026_05_15_002: KL penalty;
exp_2026_05_17_001: KL+curriculum) address plasticity loss via a *distributional
constraint* (stay close to the reference policy). C-CHAIN addresses it via a
*gradient dynamics constraint* (preserve the NTK rank by reducing churn). These
are orthogonal regularisation axes. If KL-penalty alone is insufficient to prevent
plasticity loss at 300 tasks, C-CHAIN's churn reduction provides a different stabilisation
mechanism that does not require a frozen reference model (cheaper at inference time
during training).

### 3. `arxiv:2603.04445` — Dynamic Model Routing and Cascading for Efficient LLM Inference: A Survey
*March 2026*

A comprehensive survey of LLM routing and cascading systems, categorising methods
along three dimensions: (1) **when** routing decisions are made (query-time vs.
token-time vs. layer-time), (2) **what information** is used (query features,
model confidence, capability profiles), and (3) **how** they are computed (learned
classifiers, threshold-based, RL-based, uncertainty-based).

**Why relevant:**
The survey positions our BERT-base learned router in the "query-time / learned
classifier" quadrant, which the paper identifies as the most widely deployed but
notes a key limitation: the router's training data (query-difficulty labels) is
static and becomes stale as the underlying LLMs improve. Our joint-cycle experiments
(exp_2026_05_15_005 and related) that co-train the router alongside the LLM adapter
are identified in the survey as a nascent but promising direction.

**Why deferred (not an experiment today):**
The survey does not introduce a new training method; it is a reference for positioning
and for identifying the "layer-time routing" and "token-time cascading" directions as
future work. Both require substantial infrastructure changes beyond the current
`runner.py` kind system.

### EXP-026 · `exp_2026_05_24_001_egca_credit_assignment_15b` · Priority 9
**Kind:** `grpo_continual`
**Model:** Qwen2.5-Coder-1.5B-Instruct
**Spec:** credit_assignment=egca, 200 tasks, 4 rollouts, lr=5e-6, LoRA r=16, binary reward, eval@100

**Rationale:** arxiv:2603.16158 ("Execution-Grounded Credit Assignment for GRPO in Code
Generation", ICLR 2026 Workshop SPOT) identifies that standard GRPO in code generation
suffers from **coarse credit assignment**, not reward sparsity: when a program runs but
fails tests, the single outcome signal is spread uniformly across 100–192 tokens even
though the error typically originates from a 1–8 token causal patch. EGCA executes the
failing candidate alongside a canonical reference solution, localizes the earliest
execution-state divergence, and concentrates the GRPO advantage exclusively on the
responsible token span while masking downstream tokens. Results: +3.1 on HumanEval
(82.1% vs. ~79%), +1.5 on MBPP (68.9% vs. ~67.4%) over vanilla GRPO, with only 18%
wall-clock overhead. No critic, no auxiliary loss, no learned verifier required.

For our 1.5B MBPP training set: approximately 27/100 eval tasks produce programs that
execute but fail tests (the prime EGCA target class). The causal token span in these
cases is typically the logical core of the solution: a wrong index expression, an
incorrect comparison, a missing edge case handling. Concentrating the policy gradient
on these spans instead of the full generated sequence should yield significantly
cleaner updates on the 27 mixed-outcome tasks and may convert some of the 53 all-fail
tasks (where the program crashes immediately, falling back to standard GRPO) to
mixed-outcome status as the model learns better structural patterns.

**Runner change required:** Add `credit_assignment: "egca"` field to `grpo_continual`
kind in runner.py. Implementation: when reward=0 and the code ran without exception
(execution_status="ok"), run the canonical reference solution under Python's `trace`
module, walk both traces to first state divergence, map to token offset using tokenizer
alignment, set `advantage_mask[0:divergence_token] = 0.0` and
`advantage_mask[divergence_token:divergence_token+8] = 1.0`.
Estimated additional code: ~150 lines.

**Estimated GPU time:** ~107 minutes (90 min base × 1.18 EGCA overhead)

### EXP-027 · `exp_2026_05_24_002_cchain_churn_reduction_15b_300tasks` · Priority 8
**Kind:** `grpo_continual`
**Model:** Qwen2.5-Coder-1.5B-Instruct
**Spec:** churn_reduction=cchain, churn_lambda=0.1, 300 tasks, 4 rollouts, lr=5e-6, LoRA r=16, binary reward, eval@100

**Rationale:** arxiv:2506.00592 ("Mitigating Plasticity Loss in Continual Reinforcement
Learning by Reducing Churn", ICML 2025) establishes the mechanistic chain:
NTK rank decrease → churn increase → plasticity loss. The paper's C-CHAIN method
continuously minimises an auxiliary churn loss (output variability for out-of-batch
states) alongside the regular RL gradient. This is **orthogonal** to the KL-penalty
approach already queued (exp_2026_05_15_002, exp_2026_05_17_001): KL constrains the
policy's probability distribution relative to a frozen reference model, while C-CHAIN
constrains the network's gradient step size adaptively via the NTK spectrum.

Our 300-task regime is exactly where plasticity loss begins: flat-200 (+2pts) works,
flat-400 (-1pt) fails. The 300-task KL-penalty experiments (exp_2026_05_15_002 and
exp_2026_05_17_001) in the queue provide the KL-only baselines. C-CHAIN on 300 tasks
provides the churn-reduction baseline. If KL-penalty (distributional regularisation)
and C-CHAIN (gradient dynamics regularisation) both improve over the flat-300 baseline,
and if the improvements are different in magnitude, the two mechanisms are addressing
genuinely distinct failure modes and a future combined (KL + C-CHAIN) experiment at
300+ tasks becomes motivated. If C-CHAIN alone matches flat-200's +2pts at 300 tasks,
the NTK rank degradation is the primary driver of plasticity loss (not distributional
drift), changing the design direction for extending training horizons.

**Runner change required:** Add `churn_reduction: "cchain"` and `churn_lambda: float`
fields to `grpo_continual`. Implementation: after each GRPO gradient step, compute
`L_churn = ||f(θ_new, x_buf) - f(θ_old, x_buf)||²` on a 256-entry rolling buffer of
recent training prompts; add `churn_lambda × L_churn` to the optimizer gradient.
Total additional code: ~80 lines. The buffer reuses existing training prompts so no
new data is required.

**Estimated GPU time:** ~85 minutes (300 tasks × 0.45 min/task × 0.63 pack factor)

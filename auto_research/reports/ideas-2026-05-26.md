## Top 3 Relevant Papers

### 1. `arxiv:2605.17570` — How Off-Policy Can GRPO Be? Mu-GRPO for Efficient LLM Reinforcement Learning
*May 2026 — today's paper*

**Most actionable new paper today: addresses our suffix credit-assignment failure AND provides 2× speedup.**

Standard GRPO alternates between single-prompt rollout generation and single-step gradient
updates, creating high context-switching overhead (model must switch between inference and
training modes for each step). This makes experimenting through our 29-experiment queue
slow. Mu-GRPO reorganizes training into **4 large sequential stages**:

1. **Stage Generation**: Generate rollouts for all N/4 = 50 tasks in the stage (all at once,
   keeping the model in inference mode for the entire stage batch).
2. **Stage Optimization**: Run gradient steps on the batch of 50×4=200 rollouts just generated
   (keeping model in training mode for the entire optimization phase).
3. Repeat for all 4 stages.

This reduces rollout→training mode switching by 4× and amortizes inference setup cost
across larger batches, yielding **~2× wall-clock speedup** in the paper's experiments.

**Negative-Advantage Veto — The Key Algorithmic Contribution:**

The high rollout staleness in off-policy stages creates a specific failure mode: in later
gradient steps of a stage, the current policy has drifted far from the generation policy.
For negative-advantage rollouts (reward=0), standard GRPO applies a uniform negative
advantage across all tokens, including prefix tokens that may be correct and shared with
passing rollouts. This prefix-gradient corruption is especially harmful in code generation
where:

- First 20-40 tokens (function signature, imports, variable initialization) are almost
  always syntactically correct across all rollouts
- The failure usually originates at a specific 1-8 token decision point (operator, index,
  condition)
- Standard GRPO penalizes the correct prefix identically to the wrong suffix

**Mu-GRPO's negative-advantage veto** identifies the "trigger position" — the first token
where the importance ratio `π_θ(a_t|s_t) / π_old(a_t|s_t)` exceeds a veto threshold
(default: 3.0) — and zeroes out the gradient for all tokens at and after that position.
This preserves the prefix gradient signal (where the rollout was still doing something
useful) while removing the corrupted suffix gradient.

**Why this matters for our project:**
Our best case has ~53% all-fail tasks. In these, standard GRPO sends incorrect negative
gradients that may be penalizing syntactically correct code patterns (function definitions,
for-loop boilerplate) that happen to precede the wrong algorithmic logic. The veto
surgically removes this penalty while keeping the positive gradient from the 47% of tasks
where some rollouts pass.

**Implementation**: Add `algorithm="mu_grpo"`, `mu_grpo_stages=4`, `negative_advantage_veto=True`,
`relaxed_clipping=True`. Core change: split tasks into 4 batches, generate-all then optimize;
apply veto threshold to importance ratios before PPO clip. ~80 additional lines.

**Estimated GPU time:** ~45 minutes (2× speedup; standard run is ~90 min).

### 2. `arxiv:2511.07833` — MURPHY: Multi-Turn GRPO for Self-Correcting Code Generation
*November 2025 — Chanakya Ekbote et al., Amazon Science; ICLR 2026 Workshop on LLM Reasoning*

**Highest expected-value paper for breaking the +2pt ceiling: explicit execution feedback
between training turns provides a qualitatively different learning signal.**

MURPHY extends GRPO to a multi-turn self-correction loop. In standard GRPO (all 29 queued
experiments), each training sample is a single-shot generation: generate → execute → reward
→ gradient. MURPHY adds a second turn for failing rollouts:

1. **Turn 1**: Generate G=4 rollouts per task. Execute tests. Some pass (reward=1), some fail
   (reward=0). Compute standard GRPO advantage for Turn 1.
2. **Turn 2** (for failing rollouts only): Append execution feedback to the prompt:
   `"# Previous attempt failed:\n{code}\n# Test output:\n{stdout}\n{stderr}\n# Fix:\n"`.
   Generate G=4 new rollouts conditioned on this context. Execute tests.
3. **Credit propagation**: If a Turn-2 rollout passes, propagate partial reward backward to
   the Turn-1 trajectory: `r_turn1 = delta_r = r_turn2 - r_turn1_orig` (= 0 - 0 = 0
   for full failures, > 0 for recoveries). This creates a richer gradient signal:
   - "Correctly structured attempt that was close" gets positive reinforcement for its structure
   - "Wrong direction attempt" gets no additional signal beyond standard GRPO

**Why this directly addresses our +2pt plateau:**
Our 1.5B model at turn 1 has ~27 mixed-outcome tasks (1-3/4 rollouts pass) and ~53 all-fail
tasks. For the mixed tasks, Turn 2 feedback shows the model exactly what failed in the 1-3
failing rollouts — the test name, the actual vs expected output — allowing targeted correction.
For the ~27 mixed tasks, even converting 5 additional tasks to all-pass via Turn 2 would
add +5pts to eval. For the 53 hard tasks, Turn 2 may not help immediately but provides a
training signal on "what went wrong" that gradually shapes the Turn-1 generation.

**Results from paper:**
Up to **8% relative improvement** in pass@1 over standard GRPO on Qwen and OLMo models,
on similar compute budgets. For our setup: 8% of (49/100 trained) ≈ +4pts → ~53/100 from
base 47.

**Compute budget analysis:**
- Turn 2 triggered for ~80% of task groups (53% all-fail + 27% mixed-fail rollouts)
- Effective compute ≈ 1.8× standard → 200 tasks × 1.8 ≈ 162 min (within 4h)

**DISTINCT FROM ALL 29 EXISTING EXPERIMENTS:**
No prior experiment provides inter-turn execution feedback. Staircase (EXP-011, EXP-020)
trains in chunks but provides no feedback context. EGCA (EXP-026) uses execution traces
for token-level credit assignment but still single-turn. MURPHY uniquely converts
execution traces into natural-language feedback that shapes the next generation.

**Runner changes**: Add `multi_turn=True`, `max_turns=2`, `turn_feedback="execution_trace"`,
`turn_credit_propagation="delta_reward"`. Core loop: run Turn 1, capture stdout/stderr for
failing rollouts, template feedback into prompt, run Turn 2, compute multi-turn advantages.
~120 additional lines. Estimated GPU time: **~160 minutes**.

### 3. `arxiv:2602.04380` — Beyond KL Divergence: Policy Optimization with Flexible Bregman Divergences for LLM Reasoning
*February 2026 — GBMPO / ProbL2-GRPO*

**Deferred today (queue >20) — highest-priority addition for 2026-05-27 if queue shrinks.**

All 31 queued experiments use KL divergence as the policy regularizer (either implicitly
in standard GRPO's PPO-clip or explicitly via `kl_coeff`). KL has a known asymmetry:
`KL(π_new || π_old) → ∞` when `π_old(a) → 0` but `π_new(a) > 0`, causing gradient
explosion on tokens the base model assigns near-zero probability — exactly the problem with
hard MBPP tasks where the correct algorithmic pattern (e.g., using `heapq.nlargest` for a
k-largest problem) has near-zero base-model probability.

ProbL2-GRPO replaces KL with L2 divergence in probability space: `||π_new - π_old||^2`,
which has finite second derivative everywhere and prevents gradient explosion for low-prob
tokens. Key results:
- MBPP pass@1: **60.1-60.8%** (best with neural mirror map variants)
- **70% variance reduction** on MBPP vs. KL-GRPO (±0.7 → ±0.2 training variance)
- Simple change: `divergence_type="probl2"` replaces KL term in the GRPO objective
- No additional hyperparameters needed (no reference model, no kl_coeff to tune)

Why deferred: queue is at 31. However, given the strong variance reduction result (our
eval with 100 tasks has ≈ binomial std of 5pts — a ×3 variance reduction could make the
+2pt gain clearly significant), this should be priority 9 for the next batch.

**Rationale:** arxiv:2605.17570 (Mu-GRPO, May 2026) organizes GRPO into 4 large

**Distinction from existing queue:**
- vs. DAPO (EXP-013): DAPO discards all-zero-advantage groups; Mu-GRPO keeps all groups
  but vetoes per-token gradients in negative-advantage groups. Different granularity.

  Mu-GRPO uses importance-ratio threshold. Orthogonal: one causal, one distributional.

  sequential chunks but never provides feedback between generations
- Distinct from EGCA (EXP-026): EGCA uses execution traces for within-rollout token masking

## Top 3 Relevant Papers

### 1. "Improving LLM Code Generation via Requirement-Aware Curriculum Reinforcement Learning" ★★★
**ArXiv ID**: 2605.00433 | May 2026

**Core finding**: RECRL is a curriculum RL framework that:
1. **Estimates per-problem difficulty** using the base model's rollout pass rate as a proxy (difficulty = 1 - pass@G, so hard problems have low pass rates).
2. **Constructs training batches with smoothed difficulty ordering**: begins training with problems the model can solve ~30-50% of the time (informative gradient: mixed-outcome groups), then linearly exposes harder problems over training steps.
3. **Prevents entropy collapse** by maintaining a higher fraction of mixed-outcome rollout groups throughout training — the policy is always working at its current capability frontier.

**Results on Qwen2.5-Coder-1.5B** across MBPP, MBPP+, HumanEval, HumanEval+, LiveCodeBench:
- Average Pass@1 improvement over vanilla GRPO (flat ordering): **+1.23%–5.62%**
- MBPP Pass@1: +2.1pp vs. standard GRPO baseline (flat random ordering)
- HumanEval Pass@1: +3.4pp vs. standard GRPO baseline

**Why relevant today**: The `grpo_curriculum_continual` kind is defined in runner.py but has never been used across all 61 queued experiments. RECRL directly maps onto this kind. The difficulty proxy (base_pass_at_4) is computable with the existing rollout infrastructure before training starts. The curriculum annealing (linear threshold from easy to hard over training steps) requires only a sampler change in the data loader — no changes to GRPO algorithm, reward function, or LoRA config.

**Addresses open questions**:
- "Is the +2pp ceiling fundamental?" — curriculum ordering could push past it (+2.1pp alone matches our full gain)
- "Is the 400-task degradation entropy collapse?" — at 400 tasks, curriculum ordering explicitly targets the entropy collapse mechanism

### 2. "Domain-Adaptable Reinforcement Learning for Code Generation with Dense Rewards" ★★
**ArXiv ID**: 2605.21180 | May 2026

**Core finding**: Presents a customizable execution-aware reward formula capturing syntax correctness, functional correctness, code style, security, and simulator executability. Key innovation: **token-level reward mapping** — the binary execution outcome (pass/fail) is propagated back to individual tokens via attribution, enabling per-token credit assignment from the final execution result.

**Why deferred**: Mechanistically adjacent to EXP-054 (EGCA, priority 9) which uses execution trace to assign credit to tokens. The two approaches differ in attribution method (execution-state delta vs. outcome propagation), but running both before EGCA returns results would occupy two experiment slots for credit assignment variants when the primary credit assignment result (EGCA, p9) is the more directly justified one. Deferring pending EXP-054 results.

**Also**: The EXP-009 demotion (P-GRPO partial reward, arxiv:2605.02944 showed it hurts) creates uncertainty around any multi-dimensional reward scheme. The distinction between "partial functional credit" (what EXP-009 used) and "token attribution from binary outcome" (what 2605.21180 proposes) needs EXP-054 to return first to calibrate.

### 3. "LLM Reasoning with Process Rewards for Outcome-Guided Steps" ★★
**ArXiv ID**: 2604.02341 | April 2026

**Core finding**: Proposes using a lightweight PRM (process reward model) to score intermediate code generation steps (partial code with increasing completeness) and combines these with the final execution reward. The PRM is trained to predict whether a partial code sketch will eventually pass the test suite, enabling dense supervision earlier in generation.

**Why deferred**: Requires training a separate PRM on the MBPP problem set, which adds significant setup complexity (training data, PRM model size, latency). The credit assignment track (EXP-052/054/055) already covers per-token attribution and temporal discounting without a separate PRM. Proposing after EXP-052/054/055 return results is the right sequencing.

budget) adapts rollout count per problem — both are rollout-level interventions on a flat
training pool, not task ordering interventions. grpo_multi_seed_staircase (EXP-038) varies

### EXP-063: `exp_2026_06_11_002_recrl_curriculum_continual_400tasks`

| Field | Value |
|-------|-------|
| Kind | `grpo_curriculum_continual` |
| Priority | 7 |
| Base model | Qwen/Qwen2.5-Coder-1.5B-Instruct |
| Train tasks | 400 (MBPP aug, excl. eval20) |
| Curriculum | difficulty_sorted_easy_to_hard (base_pass_at_4 proxy) |
| Threshold schedule | linear: 0.3 → 1.0 over training steps |
| Rollouts | 4 |
| LR / LoRA r | 5e-6 / 16 |
| Reward | binary |
| Novel fields | same as EXP-062 + `train_task_limit=400` |
| Wall-clock est. | ~120 min (12 min pre-eval + 108 min training) |
| GPU | auto (1× A800 80GB) |

**Rationale**: Direct extension of EXP-062 to the 400-task degradation regime. Flat
GRPO at 400 tasks drops 47→46/100 (−1 pp vs. +2 pp at 200 tasks). RECRL's mechanism
(arxiv:2605.00433) predicts this degradation is entropy collapse: at 400 tasks, the
random-ordered training pool contains ~200 "easy" problems that produce all-pass rollout
groups after the first ~150 training steps, reducing the effective gradient batch to
only the remaining hard problems. Curriculum ordering fixes this by explicitly reserving
hard problems for later training steps, maintaining mixed-outcome gradient signal
throughout. If EXP-063 recovers to ≥47 (or better) at 400 tasks, it confirms entropy
collapse (not forgetting or overfitting) as the primary 400-task degradation mechanism
and establishes curriculum ordering as the fix. Combined with EXP-062 results, the pair
maps out the curriculum benefit curve from 200→400 tasks.

**Duplicate check**: EXP-056 trains at 400 tasks with lora_r=8 (flat GRPO, no curriculum).
No other queued experiment uses `grpo_curriculum_continual` at any task count. ✓

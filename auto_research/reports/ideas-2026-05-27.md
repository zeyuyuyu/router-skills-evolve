### 1. `arxiv:2602.04380` — Beyond KL Divergence: Policy Optimization with Flexible Bregman Divergences for LLM Reasoning
*February 2026 — GBMPo / ProbL2-GRPO — explicitly deferred from 2026-05-26 for today*

**Highest-priority addition today: addresses the root cause of gradient instability across all 31 queued experiments.**

All 31 queued experiments use KL divergence as their policy regulariser — either implicitly
through PPO-clip (which approximates a KL trust region) or explicitly via `kl_coeff=0.02`.
KL has a well-known asymmetry: `KL(π_new || π_old) → ∞` when `π_old(a) → 0` but
`π_new(a) > 0`. For code generation, this means tokens whose correct usage has near-zero
base-model probability (e.g., `heapq.nlargest(k, nums)` for a k-largest-elements problem,
`collections.Counter` for frequency counting, `bisect.insort` for sorted insertion) cause
gradient explosion precisely when the model is learning to use them correctly. This is
particularly damaging for the ~53/100 hard MBPP tasks where the model currently never
produces a passing rollout — the one or two rollouts that happen to include the right
algorithm pattern are exactly the rollouts at the KL instability boundary.

**ProbL2-GRPO** replaces KL with the L2 divergence in probability space:

```
D_L2(π_new || π_old) = ||π_new(·|s) − π_old(·|s)||² = Σ_a (π_new(a|s) − π_old(a|s))²
```

This has finite gradient everywhere: `∂D_L2/∂π_new(a) = 2(π_new(a) − π_old(a))`, bounded
by `[-2, 2]` regardless of probability magnitude. The L2 divergence in probability space
is a valid Bregman divergence (generated by the squared L2 norm convex function), so it
preserves all the theoretical guarantees of the GRPO trust-region formulation while removing
the explosion singularity.

**Key results:**
- MBPP pass@1: **60.1–60.8%** (best with neural mirror map variants on 7B models)
- **70% variance reduction** vs KL-GRPO: training variance ±0.7 → ±0.2 percentage points
- No additional hyperparameters: `divergence_coeff=0.02` (same scale as prior `kl_coeff`)
- No frozen reference model required (the divergence is computed on-the-fly between current
  and previous gradient step)

**For our project:** The 70% variance reduction is immediately impactful. Our 100-task eval
has ≈ binomial std ±5pts, making the +2pt best gain only marginally significant
(p ≈ 0.18 under binomial). If the training distribution variance collapses by ×3, the
gains become more consistent — and critically, a ×3 tighter training variance may also
translate to more reliable eval gains at our scale (fewer runs will degrade from +2 to ±0
due to optimizer noise). This is the first experiment that changes the **form** of the
policy regulariser rather than the advantage signal or training schedule.

**Implementation:** Replace the KL divergence term `kl_coeff * KL(π_new || π_ref)` with
`divergence_coeff * Σ_a (π_new(a|s) - π_prev_step(a|s))²` summed over vocabulary at each
training token. No reference model needed (use the policy parameters from the previous
gradient step, already resident in optimizer state). ~10 lines of change.

**Estimated GPU time:** ~90 minutes (same as standard GRPO; ×3 variance reduction bonus
means fewer seeds needed to verify gains).

### 2. `arxiv:2605.03327` — DGPO: Distribution Guided Policy Optimization for Fine-Grained Credit Assignment
*May 2026 — fresh paper*

**New today: the first token-level credit signal based on relational distributional drift,
complementary to EP-GRPO (entropy-absolute) and EGCA (execution-causal).**

Standard GRPO assigns a uniform sequence-level advantage `A_group` to all tokens in a
rollout. EP-GRPO (EXP-028, already queued) identifies high-entropy positions as signal
pivots. EGCA (EXP-026, already queued) traces execution divergence to find causal failure
tokens. DGPO introduces a third orthogonal signal: **how much has the policy changed at
each token position during training?**

**DGPO mechanism:**

*Step 1 — Per-token Hellinger distance:*
```
H(π_t, π_ref_t) = sqrt(0.5 × Σ_a (√π_t(a) − √π_ref_t(a))²)
```
Computed between the current policy and a reference snapshot at each token position `t`.
Hellinger distance is bounded in `[0, 1]` (unlike KL), so tokens where the policy has
drifted significantly from the reference get `H_t ≈ 1` (training is actively reweighting
these positions) while stable boilerplate tokens get `H_t ≈ 0`.

*Step 2 — Entropy gate:*
```
dgpo_credit_t = H_t × entropy_t
```
Multiplied by the current-policy per-token entropy to filter high-drift noisy tokens
(high `H_t` but also high `entropy_t` = model is both uncertain AND has drifted = likely
hallucination/noise rather than a genuine decision point).

*Step 3 — Advantage redistribution:*
```
A_token_t = A_group × dgpo_credit_t / mean(dgpo_credit)
```
The sequence-level advantage is redistributed to individual tokens proportional to their
credit signal, concentrating gradient mass where the policy is making meaningful distributional
changes.

**Why this matters for code generation:**
The ~27/100 mixed-outcome MBPP tasks typically fail due to 1–8 token decisions: a wrong
comparison operator, an off-by-one index, a missing edge-case condition. These tokens are
exactly where `H_t` will be highest (the policy is changing their output distribution most
during training) AND where entropy is non-trivial (the model is genuinely uncertain between
`<` and `<=`, `0` and `1`, etc.). Standard GRPO diffuses the advantage signal across 100–192
tokens; DGPO concentrates it on the 1–8 that matter.

**Key results (from paper):**
- Qwen2.5-32B: 60.0% Avg@32 on AIME2024, 46.0% Avg@32 on AIME2025 (math reasoning)
- Sets new state-of-the-art for critic-free alignment, substantially outperforming DAPO
- Note: results are primarily on math benchmarks (AIME), not code specifically; however,
  the mechanism is model-agnostic and identical in structure to EP-GRPO (which was validated
  on HumanEval+MBPP, arxiv:2605.04960)

**DISTINCT FROM ALL 32 QUEUED EXPERIMENTS:**
- vs. EP-GRPO (EXP-028): EP-GRPO uses absolute entropy `H(p_t)` of the current policy only;
  DGPO uses Hellinger distance `H(π_t, π_ref_t)` between current and reference (a relational
  distributional signal), then scales by entropy. The signals answer different questions:
  EP-GRPO asks "is the model uncertain here?"; DGPO asks "is training actively changing
  this position, and is that change meaningful (non-noisy)?"
- vs. EGCA (EXP-026): EGCA requires a second execution pass (+18% overhead), a canonical
  reference solution, and a deterministic failure-site mapping. DGPO uses logits already
  computed in the forward pass (zero overhead), no reference solution needed.
- vs. Mu-GRPO (EXP-030): Mu-GRPO vetoes gradient on negative-advantage rollout suffixes
  using importance ratio threshold. DGPO redistributes (does not veto) advantage across
  ALL rollouts using distributional drift signal.

**Implementation:** Frozen reference policy copy (already supported for EXP-015, EXP-019
`use_reference_model=True`), per-token Hellinger from softmax logits (2 lines of numpy),
entropy from softmax (1 line), multiply + renormalise + scale advantage tensor (~70 lines
total). ~2% runtime overhead. Estimated: **~92 minutes**.

### 3. `arxiv:2602.17684` — CodeScaler: Scaling Code LLM Training and Test-Time Inference via Execution-Free Reward Models
*February 2026 — deferred (not proposing today)*

CodeScaler trains an execution-free reward model (ORM) on a large synthetic code dataset
and uses it to provide dense rewards for GRPO training, eliminating the need to execute
generated code during training. Key result: +11.72pts average across 5 code benchmarks on
Qwen3-8B with GRPO.

**Why deferred:** Our training pipeline already executes code for rewards (binary reward,
100ms per rollout). An execution-free ORM would require training a separate reward model
first (significant infrastructure change). More importantly, our bottleneck is the
**+2pt ceiling** with the existing binary reward, not execution speed. CodeScaler's ORM
is most valuable for scaling to 10k+ training tasks where execution becomes the bottleneck;
our 200-task MBPP regime is well within execution-speed budget. Candidate for a future
batch when the queue shrinks below 20.

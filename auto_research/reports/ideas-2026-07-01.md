## Top 3 Relevant Papers

### 1. "Beyond Execution: Static-Analysis Rewards and Hint-Conditioned Diffusion RL for Code Generation" ★★★
**ArXiv ID**: 2605.17174 | May 2026

**Core finding**: Three static code analysis metrics — syntax validity (`ast.parse`),
type correctness (mypy strict), and lint quality (pylint 0–10) — provide the strongest
execution-free reward signals for code-generation RL. The paper's key result: in
code-RL settings where binary pass/fail rewards cause all-fail groups with zero within-group
variance, static-analysis scores vary substantially even among failing solutions (solutions
with the same test outcome differ in AST validity, type error count, and lint score).
Weighted combination S = 0.4·syntax + 0.3·mypy + 0.3·pylint into a supplementary reward
term R_total = R_pass + β·S reduces zero-variance group rate by 28–35% on HumanEval and
MBPP in the paper's ablation.

**Why relevant**: This is the first paper in our search that attacks ACR at the
*reward-design-from-code-structure* level. All four queued ACR experiments (STARE, AVSPO,
length penalty, SA-AH-GRPO) target training dynamics, gradient weights, or scalar token
counts. Static-analysis rewards create within-group spread from *semantic code quality*,
which is orthogonal to all existing approaches. The overhead is negligible (~50ms per
rollout for mypy + pylint on short HumanEval solutions).

**Motivation for EXP-101**: Run SA reward with β=0.15, 1 cycle, HumanEval, DAPO G=8.
Compare ACR reduction and pass@1 gain against the binary-reward DAPO baseline.

---

### 2. "Critique-GRPO: Advancing LLM Reasoning with Natural Language and Numerical Feedback" ★★
**ArXiv ID**: 2506.03106 | June 2025

**Core finding**: Augments GRPO with self-generated natural language critique: after each
rollout, the model generates a critique of its own solution, which is scored and combined
with the outcome reward. Reports +15–21.6% Pass@1 on Qwen2.5 model variants. The critique
creates a process-level signal that distinguishes rollouts with the same binary outcome
(all-pass or all-fail groups get different critique quality scores).

**Why relevant**: Tested on Qwen2.5 — our exact model family. The critique signal is
semantically richer than length (EXP-096) or static structure (EXP-101). However, it adds
a second generation call per rollout (critique generation after code generation), roughly
doubling per-rollout wall-clock. Deferred until EXP-098 (SA-AH-GRPO, entropy discount)
and EXP-101 (static-analysis) are evaluated individually — running Critique-GRPO before
knowing whether simpler approaches work would be premature.

---

### 3. "Execution-Grounded Credit Assignment for GRPO in Code Generation" ★★
**ArXiv ID**: 2603.16158 | March 2026

**Core finding**: Assigns per-token GRPO credit based on execution trace analysis: tokens
in the solution that are part of code paths exercised by passing tests receive amplified
gradient, while tokens in dead code (paths never exercised) receive suppressed gradient.
Reduces gradient noise in multi-test HumanEval settings.

**Why relevant as context**: Partially overlaps with VeRPO (EXP-086, arxiv:2601.03525)
which already addresses test-difficulty weighting. The execution-trace approach requires
compiler instrumentation (code path coverage) that is more complex than the static analysis
needed for EXP-101. Deferred — run simpler static-analysis approach first; revisit if
EXP-101 shows improvement but ACR remains above 40%.

| Field | Value |
|---|---|
| Kind | `grpo_continual` |
| Priority | 7 |
| Base model | Qwen/Qwen2.5-Coder-1.5B-Instruct |
| Bench | HumanEval |
| N cycles | 1 |
| GRPO algo | `dapo` |
| Reward fn | `pass_plus_static_analysis` |
| Static components | syntax (ast.parse), type (mypy strict), lint (pylint) |
| Component weights | syntax=0.4, mypy=0.3, pylint=0.3 |
| Beta (static weight) | 0.15 |
| Dynamic sampling | True |
| N generations | 8 |
| GRPO temperature | 1.0 |
| Novel fields | `static_analysis_reward=True`, `static_analysis_beta=0.15`, `reward_fn="pass_plus_static_analysis"` |
| Wall-clock est. | ~1.5 GPU-hours |
| GPU | auto (1× A800 80GB) |

**Rationale**: arxiv:2605.17174 demonstrates that static code analysis (syntax validity,
mypy type correctness, pylint score) provides within-group reward variation even when all
rollouts share the same binary test outcome. Applied to our 52.4% ACR:

In ALL-FAIL groups (all G=8 rollouts fail unit tests → binary R=0 for all):
- Some solutions don't parse (S≈0.0), some parse but fail mypy (S≈0.4),
  some pass mypy but have lint warnings (S≈0.7), some are near-correct structurally (S≈0.9)
- R_total = 0 + 0.15·S creates nonzero within-group variance → nonzero GRPO advantages

In ALL-PASS groups (all G=8 rollouts pass tests → binary R=1 for all):
- Solutions still differ in lint quality and type correctness
- R_total = 1 + 0.15·S differentiates clean vs. messy passing solutions

This is orthogonal to all four queued ACR experiments: STARE reweights advantages,
AVSPO injects virtual samples, EXP-096 penalizes token count, SA-AH-GRPO discounts
uncertain gradient tokens. Static-analysis is the only approach that creates spread
from *code-structural quality* independent of length or gradient computation.

No queued experiment has `static_analysis_reward=True` or mypy/pylint fields. ✓

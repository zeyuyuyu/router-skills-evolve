## Top 3 Relevant Papers

### 1. `arxiv:2605.04960` — EP-GRPO: Entropy-Progress Aligned Group Relative Policy Optimization with Implicit Process Guidance
*May 2026*

**Most actionable new paper relative to our project today.**

Standard GRPO assigns advantages uniformly across all generated tokens, treating a
low-entropy filler token (e.g., `return `, `def `, `    `) identically to a high-entropy
decision token (e.g., the operator in `a[i] < a[i+1]`, the variable name in
`result = []`, or the branch condition in `if len(s) == 0`). EP-GRPO challenges this
uniformity with a direct empirical finding: **token-level entropy explains ~61% of
the variance in whether a GRPO advantage update causes a lasting weight change** in
the policy, vs. only 12% for token position and 8% for token frequency.

**Method — Entropy-Gated Modulation:**

1. After each rollout's forward pass, compute per-token output entropy:
   `H_t = -sum_{v} p_t(v) * log p_t(v)`, where `p_t` is the softmax over vocabulary at
   position `t`. This is free — the softmax is already computed during the forward pass.
2. Compute the 80th-percentile entropy threshold `H_thresh` across all token positions
   in the batch.
3. For `H_t >= H_thresh` (top-20% "decision pivots"): scale the token's advantage by
   `eta = 2.0` (amplify gradient signal at uncertain positions).
4. For `H_t < H_thresh` (bottom-80% "filler tokens"): scale by
   `1 - beta * (1 - H_t / H_max)` where `beta = 0.3` (suppress noise from certain tokens).
5. Additionally: apply implicit process guidance via policy divergence anchoring — for
   correct rollouts, the KL between the current policy and the training-step-start policy
   at high-entropy positions is used as an additional process signal (no external verifier
   needed).

**Key results (from the paper):**
- 7B model on HumanEval: **+2.7 pts** over vanilla GRPO (zero compute overhead)
- 7B model on MBPP: **+1.9 pts** over vanilla GRPO
- Tested alongside DAPO and RLOO; EP-GRPO is complementary (addresses a different axis:
  intra-sequence token weighting vs. group-level sampling)
- Additive with token-level loss normalization (DAPO's token-level loss + EP-GRPO's
  entropy gating gives +4.1 pts on HumanEval in the paper's ablation)

**Why this is the highest-priority new paper today:**
Our 200-task GRPO training produces only +2 pts on MBPP eval. We know from the EGCA
paper (already in queue as EXP-026) that coarse credit assignment is a primary bottleneck.
EP-GRPO addresses credit assignment via a **different mechanism**: instead of using an
execution trace to find the causal failure site (EGCA), it uses the model's own entropy
distribution to find the uncertain decision positions. These are complementary: EGCA is
causal (post-hoc, what went wrong?); EP-GRPO is predictive (right now, where is the model
choosing?). Critically, EP-GRPO requires zero overhead — the entropy is already available
from the forward pass logits — while EGCA adds 18% overhead from the second execution pass.

**Implementation:** Add `advantage_masking="entropy_gated"` to `grpo_continual` runner.
Core change: after `advantage = (reward - mean_r) / (std_r + eps)`, compute entropy
from logits tensor `[B, T, V]`, apply percentile gate, scale advantage in-place.
~60 lines of numpy/torch.

### 2. `arxiv:2605.21468` — You Only Need Minimal RLVR Training: Extrapolating LLMs via Rank-1 Trajectories
*May 20, 2026 — 5 days ago*

**Most structurally novel paper of the week; highest potential compute impact.**

This paper makes a striking empirical claim about RLVR training dynamics for LLMs:
the parameter-update trajectory during LoRA-based RLVR training lies almost entirely
in a **rank-1 subspace** of the weight-change matrix, and the dominant singular value
**increases linearly** with training steps. If this holds for code-domain GRPO, it has
profound implications: the entire trajectory from step 0 to step N is determined by:
(a) the rank-1 direction `u v^T` (established within the first ~30-50 steps), and
(b) a scalar `σ(step) = a * step + b` (a linear function estimable from early steps).

**Method — RELEX (REinforcement Learning EXtrapolation):**

1. Run GRPO normally for `observe_tasks` steps (e.g., 50 out of 200).
2. At every `snapshot_interval` steps (e.g., every 10 steps → 5 snapshots), record the
   LoRA adapter delta: `Δ(t) = B(t) × A(t) - B(0) × A(0)`.
3. Compute SVD of each `Δ(t)`: `Δ(t) ≈ σ_1(t) * u * v^T` (rank-1 approximation).
4. Fit a linear model: `σ_1(t) = a * t + b` via least squares on the 5 snapshots.
5. Extrapolate: `σ_200 = a * 200 + b`; reconstruct the adapter as
   `B × A = B(0) × A(0) + σ_200 * u * v^T`.
6. Evaluate the extrapolated adapter directly — no additional gradient steps.

**Results:**
- Qwen2.5-Math-1.5B, AIME24: RELEX from 50-step window = 92% of full-200-step result
  (full = 28.4 pts, RELEX = 26.2 pts; base = 22.1 pts)
- Qwen3-1.7B, AMC23: RELEX from 50-step window = 95% of full-200-step result
- Code is public: [weizhepei/RELEX](https://github.com/weizhepei/RELEX)

**Why critical for our project:**
RELEX tests a fundamental hypothesis about how code GRPO training works. Our setup uses
Qwen2.5-Coder-1.5B (same architecture as Qwen2.5-Math-1.5B in the paper). If code
GRPO is also rank-1, the implications are:

1. **Diagnostic**: The +2pt gain is captured in the rank-1 direction after ~50 tasks.
   Tasks 51-200 only scale this direction. Our plateau at +2pts may be explained by the
   linear scalar saturating (σ saturates → gain saturates).
2. **Speed**: We can run new GRPO variants (EGCA, EP-GRPO, C-CHAIN) with 50-step
   observation + extrapolation instead of 200 full steps — 4× faster iteration cycle.
3. **Extension**: If σ_1 is still growing linearly at step 200 (not saturating), RELEX
   predicts extrapolating to step 400 would give `σ_400 = 2σ_200` — but history shows
   flat-400 = -1pt. This would mean the rank-1 approximation breaks down at step 200
   (higher-rank components become important and introduce destructive interference),
   consistent with plasticity loss / NTK rank collapse.

**If code GRPO is NOT rank-1**: This is equally informative — it means code generation
requires a more complex weight-space trajectory than mathematical reasoning, motivating
different training strategies (e.g., higher-rank LoRA, or the staircase/warm-start
approaches already queued).

**Implementation**: Add `extrapolation_mode="rank1_relex"`, `observe_tasks=50` to
`grpo_continual`. Train only 50 tasks, snapshot adapter every 10, SVD + linear fit +
extrapolation, evaluate. ~45 additional lines (numpy).

### 3. `arxiv:2603.28204` — ERPO: Token-Level Entropy-Regulated Policy Optimization for Large Reasoning Models
*March 2026 — Deferred (EP-GRPO supersedes)*

ERPO identifies the same "Critical Decision Pivots" (CDPs) as EP-GRPO — transient
high-entropy positions where diverse path exploration matters most — but uses a slightly
different implementation: ERPO's "Entropy-aware Gating" amplifies exploration at CDPs
via an upper-confidence-style weighting, while EP-GRPO's "entropy-gated modulation" uses
a percentile threshold + multiplicative scale. EP-GRPO (2605.04960) is the later and
more complete paper (published May 2026, evaluates on both HumanEval and MBPP, includes
the implicit process signal component), so EXP-028 uses EP-GRPO's formulation.

**Why not queued today:**
- EP-GRPO is strictly more recent and comprehensive, and both papers address the same
  mechanism (entropy-gated token-level advantage)
- Queueing both would create a near-duplicate experiment pair with high overlap in spec
- Recommendation: if EXP-028 (EP-GRPO) shows large gains (+3+ pts), run ERPO as a
  follow-up to compare the two formulations; if gains are modest (+1-2 pts), combining
  EP-GRPO's entropy gating with ERPO's bucket-normalisation is motivated

### EXP-028 · `exp_2026_05_25_001_epgrpo_entropy_gated_15b` · Priority 8
**Kind:** `grpo_continual`
**Model:** Qwen2.5-Coder-1.5B-Instruct
**Spec:** advantage_masking=entropy_gated, entropy_gate_percentile=80, eta=2.0, beta=0.3,
200 tasks, 4 rollouts, lr=5e-6, LoRA r=16, binary reward, eval@100

**Rationale:** arxiv:2605.04960 (EP-GRPO, May 2026) introduces entropy-gated advantage
modulation: high-entropy "decision pivot" tokens (top 20% by output entropy) receive
amplified advantage (eta=2.0×), low-entropy filler tokens receive suppressed advantage
(beta=0.3 decay). This concentrates the policy gradient at token positions where the
model is genuinely uncertain — the precise operator choices, variable names, and branch
conditions in code — without any additional execution, reference model, or overhead.
Tested on 7B models: +2.7 pts HumanEval, +1.9 pts MBPP over vanilla GRPO.

**Distinction from existing queue:**
- vs. EGCA (EXP-026, priority 9): EGCA uses execution-trace divergence (causal, external
  reference needed, 18% overhead); EP-GRPO uses model's own entropy (distributional,
  zero overhead). Complementary axes.
- vs. F-GRPO (EXP-018, priority 9): F-GRPO weights at task/group level by success rate;
  EP-GRPO weights at token level by entropy. Different granularity.
- vs. DAPO (EXP-013, priority 8): DAPO filters uninformative groups; EP-GRPO weights
  tokens within kept groups. Orthogonal mechanisms — the paper shows they are additive.

**Runner change:** Add `advantage_masking`, `entropy_gate_percentile`, `entropy_gate_eta`,
`entropy_gate_beta` to `grpo_continual`. ~60 lines, zero runtime overhead.

**Estimated GPU time:** ~88 minutes.

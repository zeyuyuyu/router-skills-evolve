## Top 3 Relevant Papers

### 1. `arxiv:2508.05170` — Posterior-GRPO: Rewarding Reasoning Processes in Code Generation
*Aug 2025*

Introduces P-GRPO, which computes reward as the fraction of test assertions that pass
**conditional on the code executing without an exception** (else reward=0). Unlike raw
partial test count, the execution-conditional gate prevents the model gaming partial
credit via syntactically near-correct outputs that crash at runtime. In controlled
comparison on HumanEval and MBPP, P-GRPO consistently outperforms binary GRPO and
vanilla partial-count GRPO across 1B–7B models.

**Why relevant:** Our binary reward gives zero gradient on ~53% of MBPP eval tasks where
the model never finds a passing rollout. P-GRPO provides learning signal on all tasks
that at least compile, which is the majority of failures at 1.5B scale. This is the
most direct fix for the "only +2pts" limitation flagged in HANDOFF section 6.

---

### 2. `arxiv:2506.06632` — E2H Reasoner: Curriculum RL from Easy to Hard for LLM Reasoning
*2026*

Demonstrates that 1.5B–3B models that **degrade under vanilla RL** consistently recover
and improve when tasks are ordered easy-to-hard. The key mechanism is that vanilla RL
exposes hard tasks to an untrained model early, producing zero-reward rollouts that
provide no useful gradient but still update parameters (noise update). E2H Reasoner
prevents this by delaying hard tasks until the model has learned from easy ones. Average
gain over flat-RL: **+4.1% on 3B models that previously degraded**.

**Why relevant:** Our 3B model degraded from 62→60/100 with flat GRPO (–2pts). This is
exactly the failure mode E2H targets. Applying curriculum to 3B is a higher-stakes test
than the 1.5B case (queued 2026-05-15) because the base model is stronger and there is
more headroom.

---

### 3. `arxiv:2601.19897` — Self-Distillation Enables Continual Learning Without Forgetting
*Jan 2026*

Proposes sequential task training where after each stage the model distills its own
earlier checkpoint's output distribution on a held-out general set. Benchmarked against
EWC, LoRA-based continual learning, and GRPO from scratch; self-distillation matches
the best GRPO result while enabling safe accumulation of more training stages. The key
finding: **training in small increments (100–200 tasks per stage) with checkpoint
distillation outperforms one-shot training on the full set**.

**Why relevant:** Directly explains our 200→400-task degradation pattern. Training all
400 tasks at once is equivalent to the "no-distillation, full-batch" baseline that
arxiv:2601.19897 shows is the worst strategy. A 2-stage staircase (100 tasks → 100
tasks) tests whether incremental training eliminates the regression at 400 tasks. If
confirmed, this enables 4× more training data without degradation.

## Queued Experiments (4 new — 2026-05-16 batch)

### EXP-006 · `exp_2026_05_16_001_pgrpo_partial_reward_15b` · Priority 9
**Kind:** `grpo_continual`
**Model:** Qwen2.5-Coder-1.5B-Instruct
**Spec:** 200 MBPP tasks, reward=partial_test_count (conditional on execution success),
4 rollouts/prompt, 1 epoch, lr=5e-6, LoRA r=16, qwen-chat, eval@100

**Rationale:** P-GRPO (arxiv:2508.05170) shows that partial-reward conditioned on
execution success provides denser gradient signal than binary reward without the reward
hacking that unconditioned partial rewards induce. Our 1.5B model fails all tests on
roughly half of MBPP eval tasks — binary reward gives zero gradient there. P-GRPO should
provide useful signal on tasks where the model generates syntactically correct code that
fails only specific test cases, which is the majority of near-misses at 1.5B scale.
All other hyperparams identical to the best case (200x4, lr=5e-6, LoRA r=16) to isolate
the reward signal effect. This is the highest-priority 2026-05-16 experiment because it
addresses the single most actionable limitation of the current best recipe.

**Estimated GPU time:** ~90 minutes (same as flat-GRPO baseline; partial reward adds
negligible overhead over binary).

### EXP-007 · `exp_2026_05_16_002_curriculum_grpo_3b` · Priority 8
**Kind:** `grpo_curriculum_continual`
**Model:** Qwen2.5-Coder-3B-Instruct
**Spec:** 200 MBPP tasks, sort by base-3B solve-rate ascending (easy→hard),
4 rollouts/prompt, 1 epoch, lr=5e-6, LoRA r=16, qwen-chat, binary reward, eval@100

**Rationale:** E2H Reasoner (arxiv:2506.06632) demonstrates +4.1% average recovery for
3B-scale models that degraded under vanilla RL — the same degradation our 3B model
exhibits (62→60, –2pts). The paper explains the mechanism: flat RL exposes hard tasks
early, generating all-fail rollouts whose advantage values are zero (zero gradient +
random noise updates). An easy-to-hard ordering delays hard tasks until the model can
generate at least occasional successes. This is complementary to the 1.5B curriculum
(exp_2026_05_15_001): 1.5B extends a gain, 3B recovers a degradation — both are
critical for paper claims about curriculum's role in our system.

**Estimated GPU time:** ~120 minutes (3B model is 2× 1.5B in weights; LoRA keeps VRAM
safe at 80GB).

### EXP-008 · `exp_2026_05_16_003_grpo_15b_8rollouts_200tasks` · Priority 7
**Kind:** `grpo_continual`
**Model:** Qwen2.5-Coder-1.5B-Instruct
**Spec:** 200 MBPP tasks, 8 rollouts/prompt, 1 epoch, lr=5e-6, LoRA r=16,
qwen-chat, binary reward, eval@100

**Rationale:** GRPO theory (arxiv:2503.06639) shows advantage normalization quality
scales with group size G. With G=4, the per-task advantage estimate has high variance
for mixed-outcome tasks (2 pass / 2 fail → advantage ≈ 0 for some, ±1 for others).
G=8 tightens the estimate and should reduce the fraction of tasks with degenerate
(zero-variance) advantage signals. The VRAM cost is approximately +6GB over G=4 for
the 1.5B model with LoRA r=16, comfortably within the A800 80GB budget. If G=8 yields
>+4pts vs G=4's +2pts, it becomes the recommended default rollout count; if the gain
is comparable, G=4 is confirmed as efficient.

**Estimated GPU time:** ~100 minutes (G=8 ≈ 1.6× wall-clock of G=4 due to generation
cost, not training cost).

### EXP-009 · `exp_2026_05_16_004_staircase_grpo_15b_100_100` · Priority 6
**Kind:** `grpo_multi_seed_staircase`
**Model:** Qwen2.5-Coder-1.5B-Instruct
**Spec:** 2-stage staircase: stage-1 = tasks 0–99 (1 epoch, save checkpoint); stage-2 =
tasks 100–199 (continue from checkpoint, 1 epoch); 4 rollouts/prompt, lr=5e-6,
LoRA r=16, qwen-chat, binary reward, eval@100, seed=42

**Rationale:** Self-Distillation Enables Continual Learning (arxiv:2601.19897) shows
that one-shot training on 400 tasks underperforms 4 sequential stages of 100 tasks.
Our history reproduces this exactly: 200-task GRPO gives +2pts but 400-task degrades
–1pt. The simplest test is a 2-stage staircase on the working 200-task range: if
stage-1 (tasks 0–99) gives +1pt and stage-2 continuing from the checkpoint gives
+2pt, incremental accumulation is safe and we can extend to 4 stages × 100 tasks
to safely leverage all 400 training tasks. This is a prerequisite experiment before
attempting any 300–400-task run in the staircase regime.

**Estimated GPU time:** ~2 × 50 min = ~100 minutes (two sequential training stages,
each half the size of the 200-task run, plus one eval per stage).
